# Implementation Plan

- [x] 1. 共有基盤の構築（API共通レイヤー・UIプリミティブ）
- [x] 1.1 API 共通型とユーティリティを共有レイヤーに抽出する
  - API レスポンスの共通型（ページネーション付きレスポンス、単一レスポンス、エラー詳細、選択肢用の軽量型）を共有モジュールとして定義する
  - API エラークラスとレスポンスハンドリングユーティリティ、ベースURL解決を共有モジュールとして定義する
  - 既存のビジネスユニット機能の API クライアントを共有モジュールからの import に移行し、重複定義を削除する
  - ビジネスユニット機能の型定義から共通型を削除し、共有モジュールからの re-export に切り替える
  - 移行後、ビジネスユニット管理画面が正常に動作することを確認する
  - _Requirements: 全CRUD横断_

- [x] 1.2 (P) shadcn/ui Select コンポーネントを導入する
  - Radix UI Select の依存パッケージをインストールする
  - shadcn/ui パターンに準拠した Select コンポーネント群（トリガー、コンテンツ、アイテム、スクロールボタン等）を UI コンポーネントとして作成する
  - キーボード操作とアクセシビリティは Radix が担保するため、スタイリングとテーマ統一に集中する
  - _Requirements: 4.3, 4.4, 8.5_

- [x] 1.3 (P) DataTableToolbar を共通コンポーネントに抽出する
  - 検索入力・削除済みトグル・新規作成ボタンを含むツールバーを、リンク先やプレースホルダーを外部から設定可能な汎用コンポーネントとして作成する
  - DebouncedSearchInput は既存のものをそのまま利用する
  - 既存のビジネスユニット一覧画面のツールバーを共通版に置き換え、動作が変わらないことを確認する
  - _Requirements: 2.1, 2.3, 7.2_

- [x] 1.4 (P) DeleteConfirmDialog を共通コンポーネントに抽出する
  - エンティティ種別名（例: "案件", "ビジネスユニット"）と対象名を外部から設定可能な汎用削除確認ダイアログを作成する
  - 既存のビジネスユニット詳細画面の削除確認ダイアログを共通版に置き換え、動作が変わらないことを確認する
  - _Requirements: 3.4_

- [x] 2. 案件 feature の型・API クライアント・クエリ・ミューテーション定義
- [x] 2.1 案件の型定義と Zod バリデーションスキーマを作成する
  - 案件の API レスポンス型を定義する（案件ID、案件コード、名称、事業部コード/名、種別コード/名、開始年月、総工数、ステータス、期間月数、タイムスタンプ、削除日時）
  - ステータス定数（計画・確定）を定義する。将来の拡張を考慮し配列で管理する
  - 作成用スキーマ（必須: コード・名称・事業部・開始年月・総工数・ステータス、任意: 種別・期間月数）を定義する
  - 更新用スキーマ（全フィールド任意、少なくとも1つ必須）を定義する
  - 検索パラメータスキーマ（ページ・ページサイズ・検索キーワード・削除済み含むフラグ、デフォルト値付き）を定義する
  - API パラメータ型と入力型を Zod スキーマから導出する
  - 共通型は共有 API レイヤーから re-export する
  - _Requirements: 1.2, 2.2, 2.5, 4.2, 4.5_

- [x] 2.2 案件の API クライアント関数を作成する
  - 共有レイヤーの API エラークラスとレスポンスハンドリングを利用して、案件の CRUD 操作（一覧取得・単一取得・作成・更新・削除・復元）を実装する
  - 案件 API は数値 ID をパスパラメータとして使用する
  - FK 選択肢取得用に事業部一覧とプロジェクト種別一覧を取得する関数を追加する（アクティブなもののみ、全件取得）
  - _Requirements: 全CRUD, 8.1, 8.2_

- [x] 2.3 TanStack Query のクエリオプションと Query Key Factory を定義する
  - 案件の一覧取得と詳細取得のクエリオプションを定義する
  - FK 選択肢用のクエリオプションを定義し、API レスポンスから名前をラベル・コードを値とする選択肢形式に変換する
  - マスターデータの選択肢はキャッシュ時間を長めに設定する
  - _Requirements: 1.1, 3.1, 5.1, 8.1, 8.2, 8.5_

- [x] 2.4 CRUD ミューテーションフックを定義する
  - 作成・更新・削除・復元の各ミューテーションフックを定義する
  - 成功時に一覧キャッシュを無効化し、更新時は詳細キャッシュも無効化する
  - _Requirements: 4.6, 5.4, 3.5, 6.3_

- [x] 3. 案件 feature のコンポーネント定義
- [x] 3.1 一覧テーブルのカラム定義を作成する
  - 案件コード（詳細ページへのリンク付き）、名称、事業部名、種別名、開始年月、総工数、ステータス、作成日時、更新日時、アクションの各カラムを定義する
  - 全データカラムにソートヘッダーを設定する
  - 種別名が null の場合はダッシュで代替表示する
  - 開始年月は YYYY/MM 形式でフォーマットする
  - ステータスはバッジで「計画」「確定」を色分け表示する
  - 削除済みフィルタ有効時のみ、削除済み行に復元ボタンを表示する
  - _Requirements: 1.2, 1.3, 1.4, 1.5, 1.6, 6.1_

- [x] 3.2 案件の作成・編集共通フォームを作成する
  - TanStack Form を使用し、案件コード（テキスト）、名称（テキスト）、事業部（ドロップダウン）、種別（ドロップダウン、任意）、開始年月（YYYYMM形式）、総工数（数値）、ステータス（ドロップダウン）、期間月数（数値、任意）の入力フィールドを配置する
  - 事業部とプロジェクト種別はクエリで選択肢を動的取得し、Select コンポーネントで表示する。ローディング中・エラー時の状態を処理する
  - ステータスはステータス定数からドロップダウンで選択する
  - 各フィールドに Zod スキーマによるバリデーションを適用し、エラーをフィールド単位で表示する
  - 作成モードと編集モードを切り替え可能にし、編集モードでは案件コードを編集不可にする
  - 種別と期間月数の空値をサーバー送信時に null/undefined に変換する
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5, 5.1, 5.2, 5.3, 8.1, 8.2, 8.3, 8.4, 8.5_

- [x] 3.3 (P) RestoreConfirmDialog を作成する
  - ビジネスユニットの復元確認ダイアログと同パターンで案件用の復元確認ダイアログを作成する
  - _Requirements: 6.2_

- [x] 3.4 feature のパブリック API エクスポートを定義する
  - 型、スキーマ、クエリオプション、ミューテーションフック、コンポーネントを feature のエントリポイントからエクスポートする
  - _Requirements: 全体_

- [x] 4. ルート画面の実装
- [x] 4.1 案件一覧画面を実装する
  - URL の検索パラメータ（ページ、ページサイズ、検索キーワード、削除済み含むフラグ）を Zod でバリデーションする
  - クエリでページネーション付きの案件一覧を取得し、DataTable と共通 DataTableToolbar で表示する
  - 検索キーワード変更時にページを1にリセットする
  - 削除済み案件を半透明で視覚的に区別する
  - 復元ボタンのクリックで復元確認ダイアログを表示し、復元成功時にキャッシュを無効化する。復元エラー（409）をトーストで通知する
  - 画面タイトル「案件」と説明文を表示する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6, 2.1, 2.2, 2.3, 2.4, 2.5, 6.1, 6.2, 6.3, 6.4, 7.2_

- [x] 4.2 案件新規作成画面を実装する
  - パンくず（一覧 → 新規登録）を表示する
  - 作成モードの案件フォームを配置する
  - フォーム送信で作成 API を呼び出し、成功時に一覧へ遷移してトースト通知する
  - 案件コード重複エラー（409）と バリデーションエラー（422）をユーザーに表示する
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5, 4.6, 4.7, 4.8, 7.1_

- [x] 4.3 案件詳細画面を実装する
  - パンくず（一覧 → 案件名）を表示する
  - クエリで案件データを取得し、全フィールドを読み取り専用で表示する。事業部・種別はコードではなく名前を表示する
  - 編集ボタンと削除ボタンを配置する
  - 削除ボタンで共通 DeleteConfirmDialog を表示し、確定時にソフトデリート API を呼び出す。成功時に一覧へ遷移、参照制約エラー（409）・存在しないエラー（404）をトーストで通知する
  - 404 時の NotFound コンポーネントを設定する
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 3.5, 3.6, 7.1_

- [x] 4.4 案件編集画面を実装する
  - パンくず（一覧 → 案件名 → 編集）を表示する
  - クエリで既存データを取得し、編集モードの案件フォームにプリフィルする
  - フォーム送信で更新 API を呼び出し、成功時に詳細画面へ遷移してトースト通知する
  - 存在しないエラー（404）、重複エラー（409）、バリデーションエラー（422）をユーザーに表示する
  - _Requirements: 5.1, 5.2, 5.3, 5.4, 5.5, 5.6, 7.1_

- [x] 5. サイドバーナビゲーションとルーティング統合
  - サイドバーの「マスタ管理」グループに案件メニューを追加し、案件アイコンを設定する
  - 現在のパスに基づくアクティブ状態のハイライトが正しく動作することを確認する
  - TanStack Router のルートツリーが自動生成で案件ルートを認識することを確認する
  - _Requirements: 7.3_

- [x]* 6. テストの作成
- [x]* 6.1 (P) 案件の Zod スキーマのユニットテストを作成する
  - 作成スキーマの必須フィールド・文字数制限・YYYYMM形式・正の整数バリデーションをテストする
  - 更新スキーマの部分更新（少なくとも1フィールド必須）をテストする
  - 検索パラメータスキーマのデフォルト値と catch 動作をテストする
  - ステータス定数の値を検証する
  - _Requirements: 4.5, 2.5_

- [x]* 6.2 (P) 共有 API レイヤーのユニットテストを作成する
  - ApiError クラスのインスタンス生成と ProblemDetails プロパティを検証する
  - handleResponse の正常系（200, 204）と異常系（4xx, 5xx）の動作をテストする
  - _Requirements: 全CRUD横断_
