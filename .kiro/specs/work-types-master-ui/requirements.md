# 要件定義書

## はじめに

`work_types` マスターデータの管理画面を実装する。既存の `work-types` CRUD API（Hono バックエンド）を呼び出し、作業種類の検索・一覧表示・詳細表示・新規登録・編集・削除・復元を行うフロントエンド画面を提供する。TanStack Router によるファイルベースルーティング、TanStack Table によるデータテーブル、TanStack Query によるデータフェッチ、TanStack Form によるフォーム管理、shadcn/ui による UI コンポーネントを使用する。機能は `features/work-types/` に凝集し、ディレクトリ構成規約に準拠する。

UI/UX デザインは既存の business-units マスタ管理画面のデザイン言語を踏襲し、ゆとりのある余白・大きな角丸・控えめなシャドウ・スムーズなアニメーション・即時フィードバックといった特徴を適用する。

### 対象エンティティ

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| workTypeCode | string (1-20文字) | YES | 主キー。作業種類コード |
| name | string (1-100文字) | YES | 作業種類名 |
| displayOrder | number (0以上の整数) | NO (デフォルト0) | 表示順序 |
| color | string (#RRGGBB) | NO (null許容) | 表示カラーコード |
| createdAt | string (datetime) | - | 作成日時（読み取り専用） |
| updatedAt | string (datetime) | - | 更新日時（読み取り専用） |

## 要件

### 要件 1: 作業種類一覧画面

**目的:** 管理者として、作業種類の一覧をテーブル形式で閲覧したい。それにより、登録済みの作業種類を俯瞰できる。

#### 受け入れ基準

1. When ユーザーが作業種類一覧画面（`/master/work-types`）にアクセスした場合, the 管理画面 shall `GET /work-types` API を呼び出し、作業種類一覧を TanStack Table で表示する
2. The 管理画面 shall テーブルに作業種類コード・名称・カラー・表示順・作成日時・更新日時のカラムを表示する
3. The 管理画面 shall カラーカラムに `color` フィールドの値を色見本（カラースウォッチ）として視覚的に表示する
4. The 管理画面 shall TanStack Table のソート機能を提供し、各カラムの昇順・降順を切り替え可能にする
5. While API からデータを取得中の場合, the 管理画面 shall ローディング状態を表示する
6. If API からのデータ取得に失敗した場合, the 管理画面 shall エラーメッセージを表示する

---

### 要件 2: 作業種類検索・フィルタ

**目的:** 管理者として、作業種類を条件で絞り込みたい。それにより、目的のレコードを素早く見つけられる。

#### 受け入れ基準

1. The 管理画面 shall テーブル上部に検索入力欄を配置し、作業種類コードまたは名称でフィルタリングできる
2. When ユーザーが検索入力欄にテキストを入力した場合, the 管理画面 shall 入力内容に部分一致する行のみテーブルに表示する（クライアントサイドフィルタ）
3. The 管理画面 shall 「削除済みを含む」トグルを提供し、有効化時に `filter[includeDisabled]=true` パラメータで API を呼び出す
4. When 「削除済みを含む」トグルが有効な場合, the 管理画面 shall 削除済みレコードを視覚的に区別して表示する（行の透明度を下げ、「削除済み」ステータスバッジを表示）

---

### 要件 3: 作業種類詳細表示

**目的:** 管理者として、特定の作業種類の詳細情報を確認したい。それにより、登録内容を正確に把握できる。

#### 受け入れ基準

1. When ユーザーがテーブルの行をクリックした場合, the 管理画面 shall 詳細画面（`/master/work-types/$workTypeCode`）に遷移し、`GET /work-types/:workTypeCode` API を呼び出して詳細情報を表示する
2. The 管理画面 shall 詳細画面に作業種類コード・名称・カラー（カラースウォッチ付き）・表示順・作成日時・更新日時を表示する
3. The 管理画面 shall 詳細画面に「編集」ボタンと「削除」ボタンを配置する
4. The 管理画面 shall 詳細画面に一覧画面へ戻るナビゲーション（パンくずリストまたは戻るリンク）を提供する
5. If 指定されたコードの作業種類が存在しない場合, the 管理画面 shall 404 エラー画面を表示する

---

### 要件 4: 作業種類新規登録

**目的:** 管理者として、新しい作業種類を登録したい。それにより、間接作業の分類を拡張できる。

#### 受け入れ基準

1. When ユーザーが一覧画面の「新規登録」ボタンをクリックした場合, the 管理画面 shall 新規登録画面（`/master/work-types/new`）に遷移する
2. The 管理画面 shall TanStack Form を使用し、作業種類コード（必須・最大20文字・英数字とハイフン・アンダースコアのみ）、名称（必須・最大100文字）、表示順（任意・0以上の整数・デフォルト0）、カラー（任意・#RRGGBB 形式のカラーピッカー）の入力フォームを表示する
3. The 管理画面 shall カラーフィールドにカラーピッカー UI を提供し、視覚的にカラーを選択可能にする
4. The 管理画面 shall Zod バリデーションを使用し、入力値をリアルタイムでバリデーションする
5. When ユーザーがフォームを送信した場合, the 管理画面 shall `POST /work-types` API を呼び出して作業種類を作成する
6. When 作成が成功した場合, the 管理画面 shall 一覧画面にリダイレクトし、成功メッセージを表示する
7. If API が 409 Conflict エラーを返した場合, the 管理画面 shall 「同一コードの作業種類が既に存在します」というエラーメッセージを表示する
8. If API が 422 Unprocessable Entity エラーを返した場合, the 管理画面 shall バリデーションエラーの内容をフォームに表示する

---

### 要件 5: 作業種類編集

**目的:** 管理者として、既存の作業種類の情報を編集したい。それにより、名称や表示順・カラーを変更できる。

#### 受け入れ基準

1. When ユーザーが詳細画面の「編集」ボタンをクリックした場合, the 管理画面 shall 編集画面（`/master/work-types/$workTypeCode/edit`）に遷移し、現在の値をフォームに初期表示する
2. The 管理画面 shall 作業種類コードを読み取り専用で表示し、名称・表示順・カラーを編集可能にする
3. The 管理画面 shall Zod バリデーションを使用し、入力値をリアルタイムでバリデーションする
4. When ユーザーがフォームを送信した場合, the 管理画面 shall `PUT /work-types/:workTypeCode` API を呼び出して作業種類を更新する
5. When 更新が成功した場合, the 管理画面 shall 詳細画面にリダイレクトし、成功メッセージを表示する
6. If API が 404 Not Found エラーを返した場合, the 管理画面 shall 「作業種類が見つかりません」というエラーメッセージを表示する
7. If API が 422 Unprocessable Entity エラーを返した場合, the 管理画面 shall バリデーションエラーの内容をフォームに表示する

---

### 要件 6: 作業種類削除

**目的:** 管理者として、不要な作業種類を削除したい。それにより、使用されなくなった分類を非表示にできる。

#### 受け入れ基準

1. When ユーザーが詳細画面の「削除」ボタンをクリックした場合, the 管理画面 shall 確認ダイアログを表示する
2. When ユーザーが確認ダイアログで「削除」を選択した場合, the 管理画面 shall `DELETE /work-types/:workTypeCode` API を呼び出して論理削除を実行する
3. When 削除が成功した場合, the 管理画面 shall 一覧画面にリダイレクトし、成功メッセージを表示する
4. If API が 409 Conflict エラーを返した場合, the 管理画面 shall 「この作業種類は他のデータから参照されているため削除できません」というエラーメッセージを表示する
5. If API が 404 Not Found エラーを返した場合, the 管理画面 shall 「作業種類が見つかりません」というエラーメッセージを表示する

---

### 要件 7: 作業種類復元

**目的:** 管理者として、削除済みの作業種類を復元したい。それにより、誤って削除したデータを元に戻せる。

#### 受け入れ基準

1. While 「削除済みを含む」トグルが有効で削除済みレコードが表示されている場合, the 管理画面 shall 削除済み行に「復元」ボタンを表示する
2. When ユーザーが「復元」ボタンをクリックした場合, the 管理画面 shall 確認ダイアログを表示する
3. When ユーザーが確認ダイアログで「復元」を選択した場合, the 管理画面 shall `POST /work-types/:workTypeCode/actions/restore` API を呼び出して復元を実行する
4. When 復元が成功した場合, the 管理画面 shall テーブルを再取得し、成功メッセージを表示する
5. If API が 409 Conflict エラーを返した場合, the 管理画面 shall エラーメッセージを表示する

---

### 要件 8: ルーティングとナビゲーション

**目的:** 管理者として、URL ベースで画面を直接アクセスしたい。それにより、ブックマークや URL 共有が可能である。

#### 受け入れ基準

1. The 管理画面 shall TanStack Router のファイルベースルーティングで以下のルートを提供する: `/master/work-types`（一覧）、`/master/work-types/new`（新規登録）、`/master/work-types/$workTypeCode`（詳細）、`/master/work-types/$workTypeCode/edit`（編集）
2. The 管理画面 shall 検索条件を URL の search params として管理し、URL 共有時に同一の検索結果を再現可能にする
3. The 管理画面 shall 「削除済みを含む」トグルの状態を URL の search params として管理する

---

### 要件 9: ビジュアルデザイン

**目的:** 管理者として、視覚的に洗練された管理画面を使いたい。それにより、長時間の業務でも疲れにくく快適に操作できる。

#### 受け入れ基準

1. The 管理画面 shall 既存の business-units マスタ管理画面と統一されたデザイン言語（余白・角丸・シャドウ・アニメーション）を適用する
2. The 管理画面 shall ゆとりのあるスペーシングを採用し、コンテンツ領域に十分な余白を確保する
3. The 管理画面 shall カード・ボタン・入力フィールドに大きめの角丸を適用し、柔らかい印象を与える
4. The 管理画面 shall コンテンツ領域の最大幅を制限し、横幅が広がりすぎないようにする
5. The 管理画面 shall テーブル行のホバー時にスムーズなハイライトトランジションを適用する

---

### 要件 10: インタラクションとフィードバック

**目的:** 管理者として、操作結果が即座にわかるフィードバックを受けたい。それにより、操作の成功・失敗を迷わず確認できる。

#### 受け入れ基準

1. The 管理画面 shall 成功通知（「保存しました」「削除しました」等）を Toast コンポーネントで表示し、数秒後に自動で消える
2. The 管理画面 shall エラー通知を Toast コンポーネントで表示し、ユーザーが手動で閉じるまで保持する
3. The 管理画面 shall フォームのバリデーションエラーを各入力フィールドの直下にインラインで表示する
4. While API リクエスト実行中の場合, the 管理画面 shall 送信ボタンを無効化し、ボタン内にスピナーアイコンを表示する
5. The 管理画面 shall 削除済みレコードにステータスバッジを表示し、アクティブ・削除済みの状態を視覚的に区別する

---

### 要件 11: feature モジュール構成

**目的:** 開発者として、機能単位でコードが凝集された構成にしたい。それにより、保守性と再利用性を確保できる。

#### 受け入れ基準

1. The 管理画面 shall `features/work-types/` 配下に api・components・hooks・types・index.ts を配置し、feature モジュールとして構成する
2. The 管理画面 shall TanStack Query の queryOptions パターンを使用し、API 呼び出しを `features/work-types/api/` に集約する
3. The 管理画面 shall Zod スキーマをフロントエンド側にも定義し、フォームバリデーションに使用する
4. The 管理画面 shall `features/work-types/index.ts` で外部に公開するコンポーネント・フック・型をエクスポートする
