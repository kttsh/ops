# Implementation Plan

- [x] 1. 型定義とバリデーションスキーマの作成
- [x] 1.1 月次間接作業負荷データの Zod バリデーションスキーマと TypeScript 型を定義する
  - 年月バリデーション用の共通スキーマを定義する: 6桁数字の正規表現チェックに加え、月部分が 01〜12 の範囲であることを refine で検証する
  - データソース区分の enum バリデーションスキーマを定義する: "calculated" または "manual" の 2 値のみ許容する
  - 作成用スキーマを定義する: 事業部コード（必須・1〜20文字）、年月（必須・YYYYMM形式）、工数（必須・0以上99999999.99以下）、データソース（必須・enum）
  - 更新用スキーマを定義する: 事業部コード（任意）、年月（任意）、工数（任意）、データソース（任意）— すべてのフィールドを optional にする
  - バルク Upsert 用スキーマを定義する: items 配列（最低1件）の各アイテムに事業部コード（必須）、年月（必須）、工数（必須）、データソース（必須）を持たせる
  - DB 行型を定義する: monthly_indirect_work_load テーブルの snake_case カラムに対応する型（monthly_indirect_work_load_id, indirect_work_case_id, business_unit_code, year_month, manhour, source, created_at, updated_at）
  - API レスポンス型を定義する: camelCase に変換した全フィールドを持つ型。日時は string（ISO 8601）、manhour は number、source は string
  - _Requirements: 3.3, 4.2, 6.2, 6.3, 8.1, 8.3, 8.4, 8.5, 8.6_
  - _Contracts: monthlyIndirectWorkLoad types Service Interface_

- [x] 2. データ変換層の作成
- [x] 2.1 (P) DB 行データから API レスポンス形式への変換関数を作成する
  - DB の snake_case フィールドを camelCase の API レスポンスフィールドに変換する関数を作成する
  - 日時フィールド（created_at, updated_at）を ISO 8601 文字列に変換する
  - manhour は number 型のまま保持する（mssql ライブラリが DECIMAL を number に自動変換）
  - source はそのまま文字列で保持する
  - _Requirements: 7.3, 7.4, 7.5_
  - _Contracts: monthlyIndirectWorkLoadTransform Service Interface_

- [x] 3. データアクセス層の作成
- [x] 3.1 一覧取得・単一取得・存在確認のクエリを実装する
  - 指定された間接作業ケースIDに紐づく負荷データ一覧を事業部コード昇順・年月昇順で取得するクエリを作成する（ページネーションなし・全件返却）
  - 負荷データIDによる単一取得クエリを作成する
  - 間接作業ケースの存在確認クエリを作成する: indirect_work_cases テーブルで deleted_at IS NULL のレコードが存在するかチェックする
  - 事業部の存在確認クエリを作成する: business_units テーブルで deleted_at IS NULL のレコードが存在するかチェックする（単一・複数の両方に対応）
  - ユニーク制約チェッククエリを作成する: 同一間接作業ケースID・同一事業部コード内での年月重複を確認する。更新時には自身のIDを除外できるようにする
  - _Requirements: 1.1, 1.3, 1.4, 2.1, 2.2, 3.5, 3.6, 3.7, 4.6, 4.7_
  - _Contracts: monthlyIndirectWorkLoadData findAll, findById, indirectWorkCaseExists, businessUnitExists, businessUnitsExist, uniqueKeyExists_

- [x] 3.2 作成・更新・物理削除のクエリを実装する
  - 新規レコード挿入クエリを作成する: OUTPUT 句で自動採番IDを取得後、単一取得クエリで完全な行を返す。事業部コード、年月、工数、データソースを挿入する
  - 更新クエリを作成する: 指定されたフィールド（事業部コード、年月、工数、データソース）のみ動的に SET 句を構成し、updated_at を現在日時に更新する
  - 物理削除クエリを作成する: DELETE 文で該当レコードを削除し、削除件数が 0 より大きいかを返す
  - _Requirements: 3.1, 4.1, 4.5, 5.1_
  - _Contracts: monthlyIndirectWorkLoadData create, update, deleteById_

- [x] 3.3 バルク Upsert のクエリを実装する
  - トランザクション内で MERGE 文をループ実行する: 既存レコードは工数、データソース、更新日時を更新、新規レコードは挿入する
  - MERGE 文の条件は間接作業ケースID、事業部コード、年月の3キー複合キーで照合する
  - トランザクション完了後、該当間接作業ケースの全レコードを事業部コード昇順・年月昇順で取得して返却する
  - 失敗時はトランザクション全体をロールバックする
  - _Requirements: 6.1, 6.4, 6.8_
  - _Contracts: monthlyIndirectWorkLoadData bulkUpsert_

- [x] 4. サービス層の作成
- [x] 4.1 一覧取得と単一取得のビジネスロジックを実装する
  - 一覧取得: 親リソース（間接作業ケース）の存在確認を行い、存在しないまたは論理削除済みの場合は 404 エラーを送出する。データ層から取得した結果を変換して返却する
  - 単一取得: 指定された負荷データを取得し、存在しない場合は 404 エラーを送出する。取得したデータの間接作業ケースIDがURLの間接作業ケースIDと一致しない場合も 404 を送出する
  - _Requirements: 1.4, 2.2, 2.3_
  - _Contracts: monthlyIndirectWorkLoadService findAll, findById_

- [x] 4.2 作成と更新のビジネスロジックを実装する
  - 作成: 親リソースの存在確認を行い、存在しない場合は 404 エラーを送出する。事業部コードの存在確認を行い、存在しない場合は 422 エラーを送出する。同一間接作業ケース・同一事業部内で年月が重複していないか確認し、重複時は 409 エラーを送出する。データ層で作成後、変換層を通じてレスポンスを返却する
  - 更新: 対象データの存在確認と間接作業ケースID整合性チェックを行う。事業部コードを変更する場合は存在確認を行い、存在しない場合は 422 エラーを送出する。事業部コードまたは年月を変更する場合はユニーク制約チェック（自身を除外）を行い、重複時は 409 エラーを送出する。データ層で更新後、変換層を通じてレスポンスを返却する
  - _Requirements: 3.5, 3.6, 3.7, 4.3, 4.6, 4.7_
  - _Contracts: monthlyIndirectWorkLoadService create, update_

- [x] 4.3 物理削除のビジネスロジックを実装する
  - 対象データの存在確認を行い、存在しない場合は 404 エラーを送出する
  - 取得したデータの間接作業ケースIDがURLの間接作業ケースIDと一致しない場合は 404 を送出する
  - データ層で物理削除を実行する
  - _Requirements: 5.2, 5.3_
  - _Contracts: monthlyIndirectWorkLoadService delete_

- [x] 4.4 バルク Upsert のビジネスロジックを実装する
  - 親リソースの存在確認を行い、存在しない場合は 404 エラーを送出する
  - リクエスト配列内の事業部コード＋年月の組み合わせを Set で重複チェックし、重複がある場合は 422 エラーを送出する
  - リクエスト配列に含まれる全事業部コードの存在確認を行い、いずれかが存在しない場合は 422 エラーを送出する
  - データ層のバルク Upsert を呼び出し、結果を変換層で変換して返却する
  - _Requirements: 6.6, 6.7, 6.9_
  - _Contracts: monthlyIndirectWorkLoadService bulkUpsert_

- [x] 5. ルート層の作成とシステム統合
- [x] 5.1 CRUD エンドポイントとバルク Upsert エンドポイントを定義する
  - 一覧取得エンドポイントを定義する: パスパラメータの間接作業ケースIDを整数に変換し、サービス層に委譲して data 配列形式のレスポンスを返却する
  - 単一取得エンドポイントを定義する: 間接作業ケースIDと負荷データIDを整数に変換してサービス層に委譲する
  - 作成エンドポイントを定義する: リクエストボディをバリデーションし、サービス層で作成後、Location ヘッダとともに 201 を返却する
  - バルク Upsert エンドポイントを定義する: PUT /bulk としてリクエストボディをバリデーションし、サービス層に委譲して data 配列形式で 200 を返却する。個別更新エンドポイントより前に定義してルーティング衝突を回避する
  - 更新エンドポイントを定義する: リクエストボディをバリデーションし、サービス層で更新後 200 を返却する
  - 削除エンドポイントを定義する: サービス層で物理削除後 204 No Content を返却する
  - パスパラメータ（indirectWorkCaseId, monthlyIndirectWorkLoadId）を各ハンドラで整数に変換し、変換不能な場合は 422 エラーを送出する
  - _Requirements: 1.1, 1.2, 2.1, 3.1, 3.2, 3.4, 4.1, 4.4, 5.1, 6.1, 6.5, 7.1, 7.2, 8.1, 8.2_
  - _Contracts: monthlyIndirectWorkLoads Route API Contract_

- [x] 5.2 アプリケーションにルートをマウントする
  - エントリーポイントに月次間接作業負荷データルートをネストルートとしてマウントする（/indirect-work-cases/:indirectWorkCaseId/monthly-indirect-work-loads の階層）
  - 既存のルートマウントとの整合性を確認する
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 5.1, 6.1_

- [x] 6. テストの作成
- [x] 6.1 一覧取得と単一取得のテストを作成する
  - 一覧取得の正常系: data 配列形式のレスポンス、事業部コード昇順・年月昇順ソートを検証する
  - 単一取得の正常系: data オブジェクト形式のレスポンス、source フィールドの存在を検証する
  - 一覧取得の異常系: 存在しない/論理削除済みの間接作業ケースIDでの 404 を検証する
  - 単一取得の異常系: 存在しない負荷データID → 404、間接作業ケースID不一致 → 404 を検証する
  - _Requirements: 9.1, 9.2_

- [x] 6.2 作成・更新・削除のテストを作成する
  - 作成の正常系: 201 ステータス、Location ヘッダ、レスポンスボディ（source フィールド含む）を検証する
  - 更新の正常系: 200 ステータス、更新されたフィールド（source 含む）の反映を検証する
  - 削除の正常系: 204 No Content を検証する
  - 作成の異常系: バリデーションエラー → 422、親ケース不存在 → 404、事業部不存在 → 422、事業部コード＋年月重複 → 409 を検証する
  - 更新の異常系: 存在しないID → 404、バリデーションエラー → 422、事業部コード＋年月重複 → 409、事業部不存在 → 422 を検証する
  - 削除の異常系: 存在しないID → 404、間接作業ケースID不一致 → 404 を検証する
  - _Requirements: 9.1, 9.2_

- [x] 6.3 バルク Upsert のテストを作成する
  - 正常系: 200 ステータス、data 配列形式のレスポンス、新規作成と既存更新の混在を検証する
  - 異常系: バリデーションエラー → 422、親ケース不存在 → 404、配列内事業部コード＋年月重複 → 422、事業部不存在 → 422 を検証する
  - _Requirements: 9.1, 9.2, 9.3, 9.4_
  - _Contracts: Vitest + app.request() パターン_
