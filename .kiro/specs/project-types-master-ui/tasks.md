# Implementation Plan

- [x] 1. 案件タイプ feature モジュールの型定義・API クライアント構築
- [x] 1.1 案件タイプの型定義と Zod スキーマを作成する
  - 案件タイプのレスポンス型（コード・名称・表示順・作成日時・更新日時・削除日時）を定義する
  - ページネーション付きレスポンス型・単一レスポンス型・ProblemDetails 型を定義する（business-units と同一構造）
  - 作成用スキーマ（コード必須・最大20文字・英数字ハイフンアンダースコアのみ、名称必須・最大100文字、表示順任意・デフォルト0）を定義する
  - 更新用スキーマ（名称必須・最大100文字、表示順任意）を定義する
  - URL 検索パラメータ用スキーマ（検索文字列、削除済み含むフラグ）を `fallback` + `.default()` パターンで定義する
  - 各スキーマから TypeScript 型を `z.infer` で導出する
  - _Requirements: 4.2, 4.3, 5.3, 11.3_

- [x] 1.2 案件タイプ API クライアント関数群を実装する
  - 一覧取得（GET /project-types）、単一取得（GET /project-types/:code）、作成（POST）、更新（PUT）、削除（DELETE）、復元（POST /actions/restore）の各関数を実装する
  - 環境変数 `VITE_API_BASE_URL` からベース URL を取得する
  - `!response.ok` 時に ProblemDetails を解析して ApiError を throw するエラーハンドリングを実装する
  - 一覧取得は `includeDisabled` パラメータを `filter[includeDisabled]` クエリパラメータとして送信する
  - _Requirements: 1.1, 3.1, 4.4, 5.4, 6.2, 7.3_

- [x] 1.3 TanStack Query の queryOptions と mutation フックを定義する
  - Query Key Factory パターンで `projectTypeKeys`（all / lists / list / details / detail）を定義する
  - 一覧取得用の `projectTypesQueryOptions` を定義する
  - 単一取得用の `projectTypeQueryOptions` を定義する
  - 作成・更新・削除・復元の 4 つの mutation フックを定義する
  - 各 mutation の `onSuccess` で一覧キャッシュ（lists）を無効化する
  - 更新 mutation では詳細キャッシュ（detail）も無効化する
  - _Requirements: 1.1, 3.1, 4.4, 4.5, 5.4, 5.5, 6.2, 6.3, 7.3, 7.4_

- [x] 1.4 feature モジュールのバレルエクスポートを作成する
  - `features/project-types/index.ts` で型・スキーマ・API フック・コンポーネントをパブリック API としてエクスポートする
  - _Requirements: 11.4_

- [x] 2. テーブル関連コンポーネントの構築
- [x] 2.1 (P) カラム定義を作成する
  - 案件タイプコード（一覧リンク付き）、名称、表示順、ステータス（アクティブ/削除済みバッジ）、作成日時、更新日時、アクション（復元ボタン）のカラムを定義する
  - 主要カラムにソートヘッダーを設定する
  - 削除済みレコードにはステータスカラムで「削除済み」赤バッジ、アクティブは「アクティブ」緑バッジを表示する
  - 復元ボタンは `deletedAt` が存在する行にのみ表示する
  - _Requirements: 1.2, 2.4, 7.1, 10.5_

- [x] 2.2 (P) IME 対応のデバウンス検索入力コンポーネントを作成する
  - 300ms のデバウンス処理を実装する
  - IME 入力中（composition 中）は検索をトリガーしない制御を入れる
  - _Requirements: 2.1, 2.2_

- [x] 2.3 テーブルツールバーコンポーネントを作成する
  - 検索入力欄（デバウンス検索コンポーネントを使用）を配置する
  - 「削除済みを含む」スイッチトグルを配置し、トグル変更時に `includeDisabled` パラメータを切り替える
  - 「新規登録」ボタンを配置し、新規登録画面へ遷移する
  - _Requirements: 2.1, 2.3, 4.1_

- [x] 2.4 データテーブルコンポーネントを作成する
  - TanStack Table のインスタンスを生成し、ソート機能を有効にする
  - クライアントサイドのグローバルフィルタで検索テキストによる絞り込みを行う
  - ローディング中はスケルトン表示またはスピナーを表示する
  - データ取得失敗時はエラーメッセージを表示する
  - 削除済み行の透明度を下げて視覚的に区別する
  - テーブル行のホバー時にスムーズなハイライトトランジションを適用する
  - _Requirements: 1.1, 1.3, 1.4, 1.5, 2.2, 9.3_

- [x] 3. フォーム・ダイアログコンポーネントの構築
- [x] 3.1 (P) 新規登録・編集共通フォームコンポーネントを作成する
  - `mode: 'create' | 'edit'` で新規登録と編集を切り替える
  - 案件タイプコード入力欄（edit モードでは読み取り専用）を表示する
  - 名称入力欄（必須）と表示順入力欄（任意、デフォルト0）を表示する
  - Zod スキーマでリアルタイムバリデーションを行い、エラーをフィールド直下にインライン表示する
  - 送信中はボタンを無効化しスピナーを表示する
  - _Requirements: 4.2, 4.3, 5.2, 5.3, 10.3, 10.4_

- [x] 3.2 (P) 削除確認ダイアログコンポーネントを作成する
  - 論理削除であり復元可能であることを説明するダイアログを表示する
  - 対象の案件タイプ名を説明文に含める
  - 「削除」「キャンセル」ボタンを配置する
  - _Requirements: 6.1_

- [x] 3.3 (P) 復元確認ダイアログコンポーネントを作成する
  - 削除済みの案件タイプを復元することを確認するダイアログを表示する
  - 「復元」「キャンセル」ボタンを配置する
  - _Requirements: 7.2_

- [x] 4. ルートファイルの構築と画面接続
- [x] 4.1 案件タイプ一覧画面のルートを作成する
  - `/master/project-types` ルートを TanStack Router のファイルベースルーティングで定義する
  - URL search params（検索文字列、削除済み含むフラグ）を Zod スキーマでバリデーションする
  - API からデータを取得してデータテーブルに渡す
  - ツールバーの操作（検索・トグル・新規登録）を search params と連動させる
  - ページタイトルに「案件タイプ」を設定する
  - _Requirements: 1.1, 2.1, 2.3, 8.1, 8.2_

- [x] 4.2 案件タイプ新規登録画面のルートを作成する
  - `/master/project-types/new` ルートを定義する
  - パンくずナビゲーション（一覧 → 新規登録）を表示する
  - フォーム送信時に作成 mutation を呼び出し、成功時に一覧画面へリダイレクトして成功 Toast を表示する
  - 409 エラーは「同一コードの案件タイプが既に存在します」、422 エラーはフォームにエラー表示する
  - _Requirements: 4.1, 4.4, 4.5, 4.6, 4.7, 10.1, 10.2_

- [x] 4.3 案件タイプ詳細画面のルートを作成する
  - `/master/project-types/$projectTypeCode` ルートを定義する
  - パンくずナビゲーション（一覧 → 詳細）を表示する
  - 案件タイプのコード・名称・表示順・作成日時・更新日時をカード形式で表示する
  - 「編集」ボタンと「削除」ボタンを配置する
  - 削除ボタンクリックで削除確認ダイアログを表示し、確認後に削除 mutation を呼び出す
  - 削除成功時に一覧画面へリダイレクトして成功 Toast を表示する
  - 409 エラーは参照制約エラーメッセージ、404 エラーは「案件タイプが見つかりません」を Toast で表示する
  - 存在しないコードの場合は 404 エラー画面を表示する
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 3.5, 6.1, 6.2, 6.3, 6.4, 6.5, 10.1, 10.2_

- [x] 4.4 案件タイプ編集画面のルートを作成する
  - `/master/project-types/$projectTypeCode/edit` ルートを定義する
  - パンくずナビゲーション（一覧 → 詳細 → 編集）を表示する
  - 既存データを API から取得してフォームの初期値に設定する
  - フォーム送信時に更新 mutation を呼び出し、成功時に詳細画面へリダイレクトして成功 Toast を表示する
  - 404 エラーは「案件タイプが見つかりません」、422 エラーはフォームにエラー表示する
  - _Requirements: 5.1, 5.4, 5.5, 5.6, 5.7, 10.1, 10.2_
