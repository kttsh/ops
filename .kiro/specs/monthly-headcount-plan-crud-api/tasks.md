# Implementation Plan

- [x] 1. 型定義とバリデーションスキーマの作成
- [x] 1.1 月次人員計画データの Zod バリデーションスキーマと TypeScript 型を定義する
  - 年月バリデーション用の共通スキーマを定義する: 6桁数字の正規表現チェックに加え、月部分が 01〜12 の範囲であることを refine で検証する（既存の yearMonth スキーマがあればインポートして再利用する）
  - ビジネスユニットコードのバリデーションスキーマを定義する: 1文字以上20文字以下の文字列
  - 作成用スキーマを定義する: ビジネスユニットコード（必須）、年月（必須・YYYYMM形式）、人員数（必須・0以上の整数）
  - 更新用スキーマを定義する: ビジネスユニットコード（任意）、年月（任意・YYYYMM形式）、人員数（任意・0以上の整数）
  - バルク Upsert 用スキーマを定義する: items 配列（最低1件）の各アイテムにビジネスユニットコード（必須）、年月（必須）、人員数（必須）を持たせる
  - DB 行型を定義する: monthly_headcount_plan テーブルの snake_case カラムに対応する型（monthly_headcount_plan_id, headcount_plan_case_id, business_unit_code, year_month, headcount, created_at, updated_at）
  - API レスポンス型を定義する: camelCase に変換した全フィールドを持つ型。日時は string（ISO 8601）、headcount は number（整数）
  - _Requirements: 3.3, 4.2, 6.2, 6.3, 8.1, 8.3, 8.4, 8.5_
  - _Contracts: monthlyHeadcountPlan types Service Interface_

- [x] 2. データ変換層の作成
- [x] 2.1 (P) DB 行データから API レスポンス形式への変換関数を作成する
  - DB の snake_case フィールドを camelCase の API レスポンスフィールドに変換する関数を作成する
  - 日時フィールド（created_at, updated_at）を ISO 8601 文字列に変換する
  - headcount は number 型（整数）のまま保持する
  - _Requirements: 7.3, 7.4, 7.5_
  - _Contracts: monthlyHeadcountPlanTransform Service Interface_

- [x] 3. データアクセス層の作成
- [x] 3.1 一覧取得・単一取得・存在確認のクエリを実装する
  - 指定された人員計画ケースIDに紐づく月次人員計画データ一覧をビジネスユニットコード昇順、年月昇順で取得するクエリを作成する（ページネーションなし・全件返却）
  - ビジネスユニットコードによるオプショナルなフィルタリング条件を追加する: クエリパラメータが指定された場合のみ WHERE 条件に追加
  - 月次人員計画データIDによる単一取得クエリを作成する
  - 人員計画ケースの存在確認クエリを作成する: headcount_plan_cases テーブルで deleted_at IS NULL のレコードが存在するかチェックする
  - ビジネスユニットの存在確認クエリを作成する: business_units テーブルで deleted_at IS NULL のレコードが存在するかチェックする（単一・複数一括の両方に対応）
  - ユニーク制約チェッククエリを作成する: 同一人員計画ケースID・同一ビジネスユニットコード内での年月重複を確認する。更新時には自身のIDを除外できるようにする
  - _Requirements: 1.1, 1.3, 1.4, 1.5, 2.1, 2.2, 3.5, 3.6, 3.7, 4.6, 4.7_
  - _Contracts: monthlyHeadcountPlanData findAll, findById, headcountPlanCaseExists, businessUnitExists, businessUnitsExist, uniqueConstraintExists_

- [x] 3.2 作成・更新・物理削除のクエリを実装する
  - 新規レコード挿入クエリを作成する: OUTPUT 句で自動採番IDを取得後、単一取得クエリで完全な行を返す
  - 更新クエリを作成する: 指定されたフィールドのみ動的に SET 句を構成し、updated_at を現在日時に更新する
  - 物理削除クエリを作成する: DELETE 文で該当レコードを削除し、削除件数が 0 より大きいかを返す
  - _Requirements: 3.1, 4.1, 4.5, 5.1_
  - _Contracts: monthlyHeadcountPlanData create, update, deleteById_

- [x] 3.3 バルク Upsert のクエリを実装する
  - トランザクション内で MERGE 文をループ実行する: 既存レコードは人員数と更新日時を更新、新規レコードは挿入する
  - MERGE 文の条件は人員計画ケースID、ビジネスユニットコード、年月の3カラム複合キーで照合する
  - トランザクション完了後、該当人員計画ケースの全レコードをビジネスユニットコード昇順、年月昇順で取得して返却する
  - 失敗時はトランザクション全体をロールバックする
  - _Requirements: 6.1, 6.4, 6.8_
  - _Contracts: monthlyHeadcountPlanData bulkUpsert_

- [x] 4. サービス層の作成
- [x] 4.1 一覧取得と単一取得のビジネスロジックを実装する
  - 一覧取得: 親リソース（人員計画ケース）の存在確認を行い、存在しないまたは論理削除済みの場合は 404 エラーを送出する。データ層から取得した結果を変換して返却する。オプショナルなビジネスユニットコードフィルタをデータ層に渡す
  - 単一取得: 指定された月次人員計画データを取得し、存在しない場合は 404 エラーを送出する。取得したデータの人員計画ケースIDがURLの人員計画ケースIDと一致しない場合も 404 を送出する
  - _Requirements: 1.4, 1.5, 2.2, 2.3_
  - _Contracts: monthlyHeadcountPlanService findAll, findById_

- [x] 4.2 作成と更新のビジネスロジックを実装する
  - 作成: 親リソースの存在確認を行い、存在しない場合は 404 エラーを送出する。ビジネスユニットの存在確認を行い、存在しないまたは論理削除済みの場合は 404 エラーを送出する。同一人員計画ケース・ビジネスユニット内で年月が重複していないか確認し、重複時は 409 エラーを送出する。データ層で作成後、変換層を通じてレスポンスを返却する
  - 更新: 対象データの存在確認と人員計画ケースID整合性チェックを行う。ビジネスユニットコードを変更する場合は変更先BUの存在確認を行う。ビジネスユニットコードまたは年月を変更する場合はユニーク制約重複チェック（自身を除外）を行い、重複時は 409 エラーを送出する。データ層で更新後、変換層を通じてレスポンスを返却する
  - _Requirements: 3.5, 3.6, 3.7, 4.3, 4.6, 4.7_
  - _Contracts: monthlyHeadcountPlanService create, update_

- [x] 4.3 物理削除のビジネスロジックを実装する
  - 対象データの存在確認を行い、存在しない場合は 404 エラーを送出する
  - 取得したデータの人員計画ケースIDがURLの人員計画ケースIDと一致しない場合は 404 を送出する
  - データ層で物理削除を実行する
  - _Requirements: 5.2, 5.3_
  - _Contracts: monthlyHeadcountPlanService delete_

- [x] 4.4 バルク Upsert のビジネスロジックを実装する
  - 親リソースの存在確認を行い、存在しない場合は 404 エラーを送出する
  - リクエスト配列内のビジネスユニットコードと年月の組み合わせを Set（`${businessUnitCode}:${yearMonth}` をキー）で重複チェックし、重複がある場合は 422 エラーを送出する
  - リクエスト配列内の全ビジネスユニットコードの存在確認を一括で行い、存在しないBUがある場合は 404 エラーを送出する
  - データ層のバルク Upsert を呼び出し、結果を変換層で変換して返却する
  - _Requirements: 6.6, 6.7, 6.9_
  - _Contracts: monthlyHeadcountPlanService bulkUpsert_

- [x] 5. ルート層の作成とシステム統合
- [x] 5.1 CRUD エンドポイントとバルク Upsert エンドポイントを定義する
  - 一覧取得エンドポイントを定義する: パスパラメータの人員計画ケースIDを整数に変換し、クエリパラメータ businessUnitCode を取得してサービス層に委譲し data 配列形式のレスポンスを返却する
  - 単一取得エンドポイントを定義する: 人員計画ケースIDと月次人員計画データIDを整数に変換してサービス層に委譲する
  - 作成エンドポイントを定義する: リクエストボディをバリデーションし、サービス層で作成後、Location ヘッダとともに 201 を返却する
  - バルク Upsert エンドポイントを定義する: PUT /bulk としてリクエストボディをバリデーションし、サービス層に委譲して data 配列形式で 200 を返却する。個別更新エンドポイントより前に定義してルーティング衝突を回避する
  - 更新エンドポイントを定義する: リクエストボディをバリデーションし、サービス層で更新後 200 を返却する
  - 削除エンドポイントを定義する: サービス層で物理削除後 204 No Content を返却する
  - パスパラメータ（headcountPlanCaseId, monthlyHeadcountPlanId）を各ハンドラで整数に変換し、変換不能な場合は 422 エラーを送出する
  - _Requirements: 1.1, 1.2, 1.5, 2.1, 3.1, 3.2, 3.4, 4.1, 4.4, 5.1, 6.1, 6.5, 7.1, 7.2, 8.1, 8.2_
  - _Contracts: monthlyHeadcountPlans Route API Contract_

- [x] 5.2 アプリケーションにルートをマウントする
  - エントリーポイントに月次人員計画データルートをネストルートとしてマウントする（/headcount-plan-cases/:headcountPlanCaseId/monthly-headcount-plans の階層）
  - 既存のルートマウントとの整合性を確認する
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 5.1, 6.1_
