# Implementation Plan

- [x] 1. (P) DBスキーマ・マイグレーション・シードデータの準備
- [x] 1.1 (P) chart_view_capacity_items テーブルの作成
  - キャパシティシナリオの表示設定（表示/非表示、色）をチャートビューに紐付けて保存するテーブルを新規作成する
  - チャートビュー削除時に関連レコードが自動削除される外部キー制約を設定する
  - 既存の chart_view_project_items / chart_view_indirect_work_items と同パターンで設計する
  - migration SQL（個別クエリ）、create-tables.sql（まっさら作成用）の両方を更新する
  - create-tables.sql では DROP TABLE の追加も忘れないこと
  - _Requirements: 3_

- [x] 1.2 (P) シードデータの拡充
  - 楽観シナリオ（シナリオ2）の月次キャパシティデータを全BU（PLANT/TRANS/CO2）×36ヶ月分追加する
  - CO2の標準シナリオ（シナリオ1）の月次キャパシティデータを現在の12ヶ月から36ヶ月に延長する
  - 既存チャートビュー（PLANT全体負荷、TRANS全体負荷、CO2全体負荷）にキャパシティ設定のシードデータを追加する
  - 楽観シナリオのキャパシティは hours_per_person=162h で計算する
  - _Requirements: 6_

- [x] 2. バックエンドAPI: 型定義とデータアクセス層
- [x] 2.1 (P) チャートビューキャパシティアイテムの型定義とデータ変換
  - DBの行データ型（snake_case）、APIレスポンス型（camelCase）、リクエストバリデーション用Zodスキーマを定義する
  - bulkUpsert リクエストではキャパシティシナリオID、表示/非表示フラグ、色コード（任意）を受け付ける
  - DB行からAPIレスポンスへのフィールドマッピング（transform）を定義する
  - 既存の chartViewIndirectWorkItem / chartViewProjectItem の命名パターンに従う
  - _Requirements: 3_

- [x] 2.2 データアクセス層の実装
  - チャートビューIDに紐づくキャパシティアイテム一覧を取得する機能を実装する（scenario_name を JOIN で取得）
  - 一括アップサート機能を実装する（MERGE SQL で既存レコードの更新 + 新規レコードの挿入を行う）
  - アップサート時にリクエストに含まれないレコードを削除する完全同期ロジックを実装する
  - チャートビューの存在確認、キャパシティシナリオの一括存在確認のヘルパーメソッドを実装する
  - _Requirements: 3_

- [x] 3. バックエンドAPI: サービスとルート
- [x] 3.1 サービス層の実装
  - 一覧取得時にチャートビューの存在を確認してから取得する
  - 一括アップサート時にチャートビューの存在確認、アイテム内のシナリオID重複チェック、全シナリオIDの存在確認を行う
  - バリデーションエラー時は適切なHTTPステータスコード（404/422）でエラーを返す
  - _Requirements: 3_

- [x] 3.2 ルートエンドポイントの実装とアプリへの登録
  - チャートビューに紐づくキャパシティアイテムの一覧取得エンドポイント（GET）を実装する
  - キャパシティアイテムの一括アップサートエンドポイント（PUT /bulk）を実装する
  - Zodバリデーションミドルウェアでリクエストボディを検証する
  - アプリケーションのルート定義にマウントする
  - _Requirements: 3_

- [x] 4. (P) CAPACITY_COLORS のカラーパレット変更
  - キャパシティラインのカラーパレットを青系から暖色系（赤・オレンジ・アンバー）に変更する
  - 変更後のすべての色が案件エリアのカラーパレットのいずれとも完全一致しないことを確認する
  - キャパシティ（上限ライン）を暖色で表現し、需要（積み上げエリア）との視覚的コントラストを確保する
  - _Requirements: 5_

- [x] 5. フロントエンド: チャート表示統合
- [x] 5.1 (P) useChartData のキャパシティ色カスタマイズ対応
  - チャートデータフックのオプションにキャパシティシナリオごとの色指定を追加する
  - キャパシティラインシリーズ構築時にカスタム色が指定されていればそれを使い、未指定ならデフォルト色にフォールバックする
  - 既存のテストファイルにキャパシティ色カスタマイズのテストケースを追加する
  - _Requirements: 4_

- [x] 5.2 SidePanelSettings のキャパシティ設定を外部管理に変更
  - キャパシティシナリオの表示/非表示と色の状態をコンポーネント内部のローカルステートから親コンポーネントが渡す props に変更する
  - Switch の変更時に親のコールバックを呼び出し、ページレベルの状態を更新する
  - ColorPicker の変更時にも親のコールバックを呼び出す
  - コンポーネント内部のキャパシティ初期化ロジックを削除する（親で初期化するため）
  - _Requirements: 2_

- [x] 5.3 ワークロードページでのキャパシティ状態管理とAPIパラメータ連携
  - キャパシティシナリオの表示/非表示状態と色設定をページレベルのステートとして管理する
  - キャパシティシナリオ一覧の取得結果をもとに全シナリオをデフォルトON（表示）で初期化する
  - チェックがONのシナリオIDのみを抽出し、チャートデータ取得パラメータの capacityScenarioIds に含める
  - 全シナリオOFF時は capacityScenarioIds を空にしてキャパシティラインを表示しない
  - SidePanelSettings にキャパシティの状態とコールバックを props として渡す
  - useChartData にキャパシティの色設定を渡す
  - _Requirements: 1, 2_

- [x] 6. フロントエンド: プロファイル統合
- [x] 6.1 キャパシティアイテムAPIクライアントとクエリ定義
  - チャートビューのキャパシティアイテム一覧取得とbulkUpsertのAPI関数を実装する
  - TanStack Query のクエリキーとミューテーション定義を追加する
  - フロントエンドの型定義にキャパシティアイテムの型を追加する
  - _Requirements: 3_

- [x] 6.2 ProfileManager でのキャパシティ設定保存・復元
  - プロファイル保存時にキャパシティシナリオごとの表示/非表示と色をbulkUpsert APIで保存する
  - プロファイル読み込み時にキャパシティアイテムをプロジェクトアイテムと並列で取得し、親に通知する
  - ワークロードページのプロファイル適用ハンドラでキャパシティ設定を復元する
  - 既存プロファイル（キャパシティ設定なし）を読み込んだ場合はデフォルト表示状態を維持する（後方互換）
  - _Requirements: 3_

- [x] 7. 品質検証
  - TypeScript コンパイルエラーが発生しないことを確認する
  - 既存の全テストが通ることを確認する
  - _Requirements: 7_
