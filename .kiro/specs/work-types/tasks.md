# 実装計画

- [x] 1. 型定義とバリデーションスキーマの作成
- [x] 1.1 作業種類の Zod スキーマと TypeScript 型を定義する
  - 作成用スキーマ: workTypeCode（必須、1-20文字、英数字・ハイフン・アンダースコア）、name（必須、1-100文字）、displayOrder（任意、0以上整数、デフォルト0）、color（任意、#RRGGBB 形式または null）
  - 更新用スキーマ: name（必須）、displayOrder（任意）、color（任意、null 許容）
  - 一覧取得クエリスキーマ: ページネーション（既存の paginationQuerySchema を拡張）と論理削除済み含有フィルタ
  - DB 行型（snake_case）と API レスポンス型（camelCase）を定義する。レスポンス型には color フィールド（string | null）を含める
  - _Requirements: 1.4, 3.2, 4.2_

- [x] 1.2 (P) 作業種類のレスポンス変換関数を作成する
  - DB 行（snake_case）から API レスポンス（camelCase）への変換を実装する
  - Date 型を ISO 8601 文字列に変換する
  - color フィールドは null をそのまま通過させる
  - _Requirements: 1.4_

- [x] 2. データアクセス層の実装
- [x] 2.1 作業種類の基本クエリメソッドを実装する
  - 一覧取得: display_order 昇順ソート、OFFSET/FETCH ページネーション、論理削除フィルタの切り替え
  - 個別取得: work_type_code による検索（アクティブのみ）
  - 論理削除済みを含む個別取得: 重複チェックと復元で使用
  - すべてのクエリで parameterized query を使用し、color カラムを含めた SELECT を実行する
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 3.3_

- [x] 2.2 作業種類の変更系メソッドを実装する
  - 新規作成: workTypeCode, name, displayOrder, color を INSERT し、OUTPUT INSERTED で作成行を返却する
  - 更新: name は必須更新、displayOrder と color は指定時のみ SET 句に追加する。color が明示的に null の場合は NULL に更新する
  - 論理削除: deleted_at に現在日時を設定し、アクティブなレコードのみ対象とする
  - 復元: deleted_at を NULL に設定し、論理削除済みのレコードのみ対象とする
  - _Requirements: 3.1, 4.1, 5.1, 6.1_

- [x] 2.3 作業種類の参照整合性チェックを実装する
  - indirect_work_type_ratios テーブルに該当 work_type_code のレコードが存在するかを確認する
  - 物理削除テーブルのため deleted_at フィルタは不要
  - _Requirements: 5.3_

- [x] 3. サービス層の実装
- [x] 3.1 作業種類の参照系ビジネスロジックを実装する
  - 一覧取得: データ層から取得した結果をレスポンス変換して返却する
  - 個別取得: 存在しない場合は 404 エラーをスローする
  - _Requirements: 1.1, 2.1, 2.2_

- [x] 3.2 作業種類の作成・更新ビジネスロジックを実装する
  - 作成: 論理削除済みを含めた重複チェックを行い、重複時は 409 エラーをスローする。論理削除済みの場合は復元を促すメッセージを含める
  - 更新: データ層の更新結果が存在しない場合は 404 エラーをスローする
  - _Requirements: 3.1, 3.3, 4.1, 4.3_

- [x] 3.3 作業種類の削除・復元ビジネスロジックを実装する
  - 削除: 参照整合性チェックを先に実行し、参照がある場合は 409 エラーをスローする。論理削除結果が存在しない場合は 404 エラーをスローする
  - 復元: 論理削除済みを含む検索で存在しない場合は 404、アクティブ状態の場合は 409 エラーをスローする
  - _Requirements: 5.1, 5.2, 5.3, 6.1, 6.2, 6.3_

- [x] 4. ルート層の実装とシステム統合
- [x] 4.1 作業種類の全 HTTP エンドポイントを定義する
  - GET /: 一覧取得（クエリバリデーション付き）、ページネーションメタ情報を含むレスポンス
  - GET /:workTypeCode: 個別取得
  - POST /: 新規作成（ボディバリデーション付き）、201 ステータスと Location ヘッダの設定
  - PUT /:workTypeCode: 更新（ボディバリデーション付き）
  - DELETE /:workTypeCode: 論理削除、204 No Content
  - POST /:workTypeCode/actions/restore: 復元
  - Hono メソッドチェーンパターンで定義し、ルート型をエクスポートする
  - _Requirements: 1.1, 1.2, 1.3, 1.5, 2.1, 2.2, 3.1, 3.4, 4.1, 4.4, 5.1, 5.2, 6.1_

- [x] 4.2 アプリケーションのルートマウントに作業種類ルートを登録する
  - メインアプリケーションに /work-types パスでルートを登録する
  - 既存のグローバルエラーハンドラが RFC 9457 形式に変換することを確認する
  - _Requirements: 1.5, 3.4, 4.4_

- [x] 5. テストの実装
- [x] 5.1 型定義とバリデーションスキーマのテストを作成する
  - 作成スキーマ: 有効な入力、必須フィールド欠落、workTypeCode の正規表現不一致、color の #RRGGBB 正規表現
  - 更新スキーマ: 有効な入力、name 欠落、color の null 許容と正規表現
  - クエリスキーマ: デフォルト値、ページサイズ上限、includeDisabled のブール変換
  - _Requirements: 3.2, 4.2_

- [x] 5.2 (P) レスポンス変換のテストを作成する
  - snake_case から camelCase への正確な変換を検証する
  - Date 型が ISO 8601 文字列に変換されることを確認する
  - color が null の場合と値がある場合の両方を検証する
  - _Requirements: 1.4_

- [x] 5.3 サービス層のテストを作成する
  - 一覧取得: データ層の呼び出しと変換の連携
  - 個別取得: 正常系と 404 エラー
  - 作成: 正常系、重複による 409（アクティブ/論理削除済み別メッセージ）
  - 更新: 正常系と 404 エラー
  - 削除: 正常系、参照ありの 409、存在しない場合の 404
  - 復元: 正常系、存在しない場合の 404、アクティブ状態の 409
  - _Requirements: 1.1, 2.1, 2.2, 3.1, 3.3, 4.1, 4.3, 5.1, 5.2, 5.3, 6.1, 6.2, 6.3_

- [x] 5.4 ルート層の統合テストを作成する
  - 全エンドポイントの HTTP ステータスコードとレスポンス形式を検証する
  - POST 作成時の 201 ステータスと Location ヘッダの正確性
  - DELETE 時の 204 No Content
  - バリデーションエラー時の 422 RFC 9457 形式レスポンス
  - Service モックによる 404, 409 エラーケースの検証
  - _Requirements: 1.1, 1.2, 1.5, 2.1, 2.2, 3.1, 3.3, 3.4, 4.1, 4.3, 4.4, 5.1, 5.2, 5.3, 6.1, 6.2, 6.3_
