# Implementation Plan

- [x] 1. 型定義とバリデーションスキーマの作成
- [x] 1.1 案件負荷データの Zod バリデーションスキーマと TypeScript 型を定義する
  - 年月バリデーション用の共通スキーマを定義する: 6桁数字の正規表現チェックに加え、月部分が 01〜12 の範囲であることを refine で検証する
  - 作成用スキーマを定義する: 年月（必須・YYYYMM形式）、工数（必須・0以上99999999.99以下の数値）
  - 更新用スキーマを定義する: 年月（任意・YYYYMM形式）、工数（任意・0以上99999999.99以下の数値）
  - バルク Upsert 用スキーマを定義する: items 配列（最低1件）の各アイテムに年月（必須）と工数（必須）を持たせる
  - DB 行型を定義する: project_load テーブルの snake_case カラムに対応する型（project_load_id, project_case_id, year_month, manhour, created_at, updated_at）
  - API レスポンス型を定義する: camelCase に変換した全フィールドを持つ型。日時は string（ISO 8601）、manhour は number
  - _Requirements: 3.3, 4.2, 6.2, 6.3, 8.1, 8.3, 8.4_
  - _Contracts: projectLoad types Service Interface_

- [x] 2. データ変換層の作成
- [x] 2.1 (P) DB 行データから API レスポンス形式への変換関数を作成する
  - DB の snake_case フィールドを camelCase の API レスポンスフィールドに変換する関数を作成する
  - 日時フィールド（created_at, updated_at）を ISO 8601 文字列に変換する
  - manhour は number 型のまま保持する（mssql ライブラリが DECIMAL を number に自動変換）
  - _Requirements: 7.3, 7.4, 7.5_
  - _Contracts: projectLoadTransform Service Interface_

- [x] 3. データアクセス層の作成
- [x] 3.1 一覧取得・単一取得・存在確認のクエリを実装する
  - 指定された案件ケースIDに紐づく負荷データ一覧を年月昇順で取得するクエリを作成する（ページネーションなし・全件返却）
  - 負荷データIDによる単一取得クエリを作成する
  - 案件ケースの存在確認クエリを作成する: project_cases テーブルで deleted_at IS NULL のレコードが存在するかチェックする
  - 年月ユニーク制約チェッククエリを作成する: 同一案件ケースID内での年月重複を確認する。更新時には自身のIDを除外できるようにする
  - _Requirements: 1.1, 1.3, 1.4, 2.1, 2.2, 3.5, 3.6, 4.6_
  - _Contracts: projectLoadData findAll, findById, projectCaseExists, yearMonthExists_

- [x] 3.2 作成・更新・物理削除のクエリを実装する
  - 新規レコード挿入クエリを作成する: OUTPUT 句で自動採番IDを取得後、単一取得クエリで完全な行を返す
  - 更新クエリを作成する: 指定されたフィールドのみ動的に SET 句を構成し、updated_at を現在日時に更新する
  - 物理削除クエリを作成する: DELETE 文で該当レコードを削除し、削除件数が 0 より大きいかを返す
  - _Requirements: 3.1, 4.1, 4.5, 5.1_
  - _Contracts: projectLoadData create, update, deleteById_

- [x] 3.3 バルク Upsert のクエリを実装する
  - トランザクション内で MERGE 文をループ実行する: 既存レコードは工数と更新日時を更新、新規レコードは挿入する
  - MERGE 文の条件は案件ケースIDと年月の複合キーで照合する
  - トランザクション完了後、該当案件ケースの全レコードを年月昇順で取得して返却する
  - 失敗時はトランザクション全体をロールバックする
  - _Requirements: 6.1, 6.4, 6.8_
  - _Contracts: projectLoadData bulkUpsert_

- [x] 4. サービス層の作成
- [x] 4.1 一覧取得と単一取得のビジネスロジックを実装する
  - 一覧取得: 親リソース（案件ケース）の存在確認を行い、存在しないまたは論理削除済みの場合は 404 エラーを送出する。データ層から取得した結果を変換して返却する
  - 単一取得: 指定された負荷データを取得し、存在しない場合は 404 エラーを送出する。取得したデータの案件ケースIDがURLの案件ケースIDと一致しない場合も 404 を送出する
  - _Requirements: 1.4, 2.2, 2.3_
  - _Contracts: projectLoadService findAll, findById_

- [x] 4.2 作成と更新のビジネスロジックを実装する
  - 作成: 親リソースの存在確認を行い、存在しない場合は 404 エラーを送出する。同一案件ケース内で年月が重複していないか確認し、重複時は 409 エラーを送出する。データ層で作成後、変換層を通じてレスポンスを返却する
  - 更新: 対象データの存在確認と案件ケースID整合性チェックを行う。年月を変更する場合は重複チェック（自身を除外）を行い、重複時は 409 エラーを送出する。データ層で更新後、変換層を通じてレスポンスを返却する
  - _Requirements: 3.5, 3.6, 4.3, 4.6_
  - _Contracts: projectLoadService create, update_

- [x] 4.3 物理削除のビジネスロジックを実装する
  - 対象データの存在確認を行い、存在しない場合は 404 エラーを送出する
  - 取得したデータの案件ケースIDがURLの案件ケースIDと一致しない場合は 404 を送出する
  - データ層で物理削除を実行する
  - _Requirements: 5.2, 5.3_
  - _Contracts: projectLoadService delete_

- [x] 4.4 バルク Upsert のビジネスロジックを実装する
  - 親リソースの存在確認を行い、存在しない場合は 404 エラーを送出する
  - リクエスト配列内の年月を Set で重複チェックし、重複がある場合は 422 エラーを送出する
  - データ層のバルク Upsert を呼び出し、結果を変換層で変換して返却する
  - _Requirements: 6.6, 6.7_
  - _Contracts: projectLoadService bulkUpsert_

- [x] 5. ルート層の作成とシステム統合
- [x] 5.1 CRUD エンドポイントとバルク Upsert エンドポイントを定義する
  - 一覧取得エンドポイントを定義する: パスパラメータの案件ケースIDを整数に変換し、サービス層に委譲して data 配列形式のレスポンスを返却する
  - 単一取得エンドポイントを定義する: 案件ケースIDと負荷データIDを整数に変換してサービス層に委譲する
  - 作成エンドポイントを定義する: リクエストボディをバリデーションし、サービス層で作成後、Location ヘッダとともに 201 を返却する
  - バルク Upsert エンドポイントを定義する: PUT /bulk としてリクエストボディをバリデーションし、サービス層に委譲して data 配列形式で 200 を返却する。個別更新エンドポイントより前に定義してルーティング衝突を回避する
  - 更新エンドポイントを定義する: リクエストボディをバリデーションし、サービス層で更新後 200 を返却する
  - 削除エンドポイントを定義する: サービス層で物理削除後 204 No Content を返却する
  - パスパラメータ（projectCaseId, projectLoadId）を各ハンドラで整数に変換し、変換不能な場合は 422 エラーを送出する
  - _Requirements: 1.1, 1.2, 2.1, 3.1, 3.2, 3.4, 4.1, 4.4, 5.1, 6.1, 6.5, 7.1, 7.2, 8.1, 8.2_
  - _Contracts: projectLoads Route API Contract_

- [x] 5.2 アプリケーションにルートをマウントする
  - エントリーポイントに案件負荷データルートをネストルートとしてマウントする（/project-cases/:projectCaseId/project-loads の階層）
  - 既存のルートマウントとの整合性を確認する
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 5.1, 6.1_

- [x] 6. テストの作成
- [x] 6.1 一覧取得と単一取得のテストを作成する
  - 一覧取得の正常系: data 配列形式のレスポンス、year_month 昇順ソートを検証する
  - 単一取得の正常系: data オブジェクト形式のレスポンスを検証する
  - 一覧取得の異常系: 存在しない/論理削除済みの案件ケースIDでの 404 を検証する
  - 単一取得の異常系: 存在しない負荷データID → 404、案件ケースID不一致 → 404 を検証する
  - _Requirements: 9.1, 9.2_

- [x] 6.2 作成・更新・削除のテストを作成する
  - 作成の正常系: 201 ステータス、Location ヘッダ、レスポンスボディを検証する
  - 更新の正常系: 200 ステータス、更新されたフィールドの反映を検証する
  - 削除の正常系: 204 No Content を検証する
  - 作成の異常系: バリデーションエラー → 422、親ケース不存在 → 404、年月重複 → 409 を検証する
  - 更新の異常系: 存在しないID → 404、バリデーションエラー → 422、年月重複 → 409 を検証する
  - 削除の異常系: 存在しないID → 404、案件ケースID不一致 → 404 を検証する
  - _Requirements: 9.1, 9.2_

- [x] 6.3 バルク Upsert のテストを作成する
  - 正常系: 200 ステータス、data 配列形式のレスポンス、新規作成と既存更新の混在を検証する
  - 異常系: バリデーションエラー → 422、親ケース不存在 → 404、配列内年月重複 → 422 を検証する
  - _Requirements: 9.1, 9.2, 9.3, 9.4_
  - _Contracts: Vitest + app.request() パターン_
