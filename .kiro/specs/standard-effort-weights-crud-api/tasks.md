# Implementation Plan

- [ ] 1. Zod スキーマと型定義の作成
- [ ] 1.1 (P) 標準工数ウェイトのリクエストスキーマと型を定義する
  - 個別追加用スキーマ: progressRate（0〜100 の整数、必須）、weight（0 以上の整数、必須）を定義する
  - 個別更新用スキーマ: weight（0 以上の整数、必須）のみを受け付ける
  - バルク upsert 用スキーマ: items 配列（0件以上、空配列許可）で各要素は progressRate と weight を含む
  - DB 行型（snake_case）: standard_effort_weight_id, standard_effort_id, progress_rate, weight, created_at, updated_at
  - API レスポンス型（camelCase）: 同等フィールドで日時は文字列型
  - _Requirements: 6.4, 6.5, 7.3, 7.4, 7.5_

- [ ] 1.2 (P) DB 行から API レスポンスへの変換関数を実装する
  - snake_case フィールドを camelCase に変換する
  - Date 型を ISO 8601 文字列に変換する
  - _Requirements: 6.4, 6.5_

- [ ] 2. データアクセス層の実装
- [ ] 2.1 基本的な CRUD クエリ関数を実装する
  - 指定された標準工数マスタに紐づく全ウェイトレコードを進捗率の昇順で取得する関数
  - ID 指定で単一レコードを取得する関数
  - 新規レコードを INSERT し、OUTPUT 句で作成結果を返却する関数
  - weight フィールドのみを UPDATE し、OUTPUT 句で更新結果を返却する関数
  - ID 指定で物理削除する関数（影響行数で成否を判定）
  - _Requirements: 1.1, 3.1, 4.1, 5.1_

- [ ] 2.2 存在確認・重複チェック関数を実装する
  - 標準工数マスタの存在確認（deleted_at IS NULL 条件付き）
  - 2カラム複合キー（マスタ ID + 進捗率）の重複チェック（更新時の除外 ID 対応付き）
  - _Requirements: 1.2, 1.3, 3.4, 3.5_

- [ ] 2.3 バルク upsert 関数を実装する
  - SQL Server の MERGE 文を使用し、マスタ ID + 進捗率で一致判定する
  - 一致時は weight と updated_at を更新、不一致時は新規挿入する
  - MERGE 後に items に含まれない進捗率のレコードを DELETE する
  - items が空配列の場合は該当マスタの全ウェイトを DELETE する
  - トランザクション内で全件を処理し、失敗時はロールバックする
  - 処理完了後に該当マスタの全レコードを進捗率昇順で返却する
  - _Requirements: 2.1, 2.2, 2.5, 2.7_

- [ ] 3. サービス層の実装
- [ ] 3.1 一覧取得のビジネスロジックを実装する
  - 親マスタの存在を確認し、存在しない・論理削除済みの場合は 404 を返す
  - 全レコードを取得して変換する。0件の場合は空配列を返す
  - _Requirements: 1.1, 1.2, 1.3, 1.4_

- [ ] 3.2 個別追加のビジネスロジックを実装する
  - 親マスタの存在確認（404）
  - 同一マスタ ID + 進捗率の重複チェック（409）
  - チェック通過後にレコードを作成し、変換して返却する
  - _Requirements: 3.1, 3.2, 3.4, 3.5, 3.6_

- [ ] 3.3 個別更新のビジネスロジックを実装する
  - 親マスタの存在確認（404）
  - 対象レコードの存在と親マスタとの紐付き確認（404）
  - レコードを更新し、変換して返却する
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5_

- [ ] 3.4 (P) 個別削除のビジネスロジックを実装する
  - 親マスタの存在確認（404）
  - 対象レコードの存在と親マスタとの紐付き確認（404）
  - 物理削除を実行する
  - _Requirements: 5.1, 5.2, 5.3_

- [ ] 3.5 バルク upsert のビジネスロジックを実装する
  - 親マスタの存在確認（404）
  - リクエスト内の重複検出: 同一 progressRate が複数ある場合は 422 を返す
  - Data 層のバルク upsert を呼び出し、結果を変換して返却する
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.7_

- [ ] 4. ルート層の実装とシステム統合
- [ ] 4.1 HTTP エンドポイントを定義する
  - GET /（一覧取得）: 親マスタ ID をパスパラメータから取得し、Service の一覧取得を呼び出して 200 で返却
  - PUT /bulk（バルク upsert）: /bulk を個別 ID ルートより前に定義し、Zod バリデーションミドルウェアを適用して 200 で返却
  - POST /（個別追加）: Zod バリデーションミドルウェアを適用し、Service の作成を呼び出して 201 + Location ヘッダで返却
  - PUT /:standardEffortWeightId（個別更新）: Zod バリデーションミドルウェアを適用し、Service の更新を呼び出して 200 で返却
  - DELETE /:standardEffortWeightId（個別削除）: Service の削除を呼び出して 204 で返却
  - パスパラメータ（standardEffortId, standardEffortWeightId）を正の整数として検証するヘルパーを使用する
  - _Requirements: 1.1, 2.1, 3.1, 3.3, 4.1, 5.1, 6.1, 6.2, 6.3, 7.1, 7.2_

- [ ] 4.2 アプリケーションのルート登録に新規エンドポイントを追加する
  - `/standard-effort-masters/:standardEffortId/weights` パスでルートをマウントする
  - 既存のルート登録パターンに従い、import と route 呼び出しを追加する
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 5.1_

- [ ] 5. テストの実装
- [ ] 5.1 (P) 型定義・スキーマと変換関数のテストを作成する
  - 各 Zod スキーマで正常値、境界値（progressRate: 0, 100）、不正値（負数、101以上、小数）のバリデーション結果を検証する
  - バルク upsert スキーマで空配列が許可されることを検証する
  - 変換関数の snake_case → camelCase 変換と ISO 8601 日時変換を検証する
  - _Requirements: 8.1, 8.2, 8.3, 8.4_

- [ ] 5.2 (P) データアクセス層のテストを作成する
  - 一覧取得の SQL 実行と進捗率昇順ソートを検証する
  - 作成・更新・削除の各クエリのパラメータ化を検証する
  - バルク upsert のトランザクション使用（begin / commit / rollback）と MERGE 文の実行を検証する
  - 空配列での全削除動作を検証する
  - 親マスタ存在チェック（deleted_at IS NULL 条件）と複合キー重複チェックを検証する
  - _Requirements: 8.1, 8.2, 8.3_

- [ ] 5.3 サービス層のテストを作成する
  - Data 層をモックし、各メソッドの正常系・異常系を検証する
  - 親マスタ不在時の 404、重複時の 409、バリデーション失敗時の 422 を含める
  - バルク upsert 内の progressRate 重複検出ロジックを検証する
  - 親所有権の不一致による 404 を検証する
  - _Requirements: 8.1, 8.2, 8.3_

- [ ] 5.4 ルート層の結合テストを作成する
  - Hono の app.request() で各エンドポイントのステータスコードとレスポンス構造を検証する
  - POST 時の 201 ステータスと Location ヘッダ、DELETE 時の 204 と空ボディを確認する
  - PUT /bulk の正常系（全置換）と空配列（全削除）を検証する
  - パスパラメータの不正値に対する 422 レスポンスを検証する
  - _Requirements: 8.1, 8.2, 8.3, 8.4_
