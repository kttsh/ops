# Implementation Plan

- [x] 1. 型定義とバリデーションスキーマの作成
- [x] 1.1 Zod スキーマと TypeScript 型を定義する
  - `targetType` の有効値（`project`, `indirect_work` 等）を列挙型で定義する
  - `colorCode` を `#` + 6桁16進数にマッチする正規表現でバリデーションする
  - 作成用スキーマ（`targetType`, `targetId`, `colorCode` すべて必須）を定義する
  - 更新用スキーマ（全フィールドオプショナル）を定義する
  - 一括 Upsert 用スキーマ（`items` 配列、各要素は作成用スキーマと同一）を定義する
  - 一覧取得クエリスキーマを既存のページネーションスキーマを拡張して定義し、`filter[targetType]` をオプショナルフィルタとして追加する
  - DB 行型（snake_case）と API レスポンス型（camelCase）を分離して定義する
  - _Requirements: 7.1, 7.3, 7.4_

- [x] 1.2 (P) Transform 関数を作成する
  - DB 行から API レスポンスへの変換関数を実装する
  - snake_case のカラム名を camelCase のレスポンスフィールドに変換する
  - `Date` 型を ISO 8601 文字列に変換する
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 6.2_

- [x] 2. Data 層の実装（基本 CRUD）
- [x] 2.1 一覧取得と個別取得のクエリを実装する
  - `BASE_SELECT` 定数でカラム一覧を一元管理する
  - ページネーション（`OFFSET ... FETCH NEXT`）とカウントクエリを実装する
  - `target_type` によるフィルタリング条件を動的に組み立てる
  - ID 指定による単一レコード取得を実装する（`deleted_at` 条件は不要）
  - パラメタライズドクエリで SQL インジェクションを防止する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 2.1_

- [x] 2.2 作成・更新・削除のクエリを実装する
  - INSERT 文を `OUTPUT INSERTED` 付きで実装し、作成結果を返却する
  - 動的 SET 句による部分更新を実装する（指定されたフィールドのみ更新）
  - `DELETE FROM` による物理削除を実装する（ソフトデリートではない）
  - ユニーク制約チェック用のクエリ（`target_type` + `target_id` で検索）を実装する
  - 更新時の重複チェック用クエリ（自身を除外した検索）を実装する
  - _Requirements: 3.1, 4.1, 5.1_

- [x] 2.3 一括 Upsert のクエリを実装する
  - SQL Server の `MERGE` 文を使用して `target_type` + `target_id` をマッチキーとした Upsert を実装する
  - トランザクション内で実行し、エラー時は全体をロールバックする
  - `OUTPUT` 句で Upsert 結果を返却する
  - _Requirements: 6.1, 6.4_

- [x] 3. Service 層の実装
- [x] 3.1 一覧取得・個別取得のビジネスロジックを実装する
  - Data 層から取得した結果を Transform 関数で API レスポンス形式に変換する
  - ページネーション情報（件数と変換済みデータ）を返却する
  - 個別取得で対象が見つからない場合は 404 エラーを throw する
  - _Requirements: 1.1, 1.5, 2.1, 2.2_

- [x] 3.2 作成・更新のビジネスロジックを実装する
  - 作成時に `target_type` + `target_id` の重複チェックを行い、重複があれば 409 エラーを throw する
  - 更新時に対象レコードの存在チェック（404）と、変更後の `target_type` + `target_id` の重複チェック（409、自身を除外）を行う
  - 成功時は Transform 関数で変換した API レスポンスを返却する
  - _Requirements: 3.1, 3.5, 4.1, 4.3, 4.4_

- [x] 3.3 削除と一括 Upsert のビジネスロジックを実装する
  - 削除時に対象レコードの存在チェックを行い、見つからなければ 404 エラーを throw する
  - Data 層の物理削除メソッドを呼び出す
  - 一括 Upsert では Data 層のトランザクション付き MERGE を呼び出し、結果を Transform で変換して返却する
  - _Requirements: 5.1, 5.2, 6.1, 6.2, 6.3_

- [x] 4. Route 層の実装とシステム統合
- [x] 4.1 CRUD エンドポイントを定義する
  - Hono メソッドチェインで GET（一覧）、GET（個別）、POST、PUT、DELETE の各エンドポイントを定義する
  - `validate()` ヘルパーでリクエストボディ・クエリパラメータに Zod バリデーションを適用する
  - POST 成功時は 201 ステータスと `Location` ヘッダーを返却する
  - DELETE 成功時は 204 ステータスを返却する
  - `/bulk` エンドポイントは `/:id` より前に定義し、ルートマッチングの優先順位を確保する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 2.1, 3.1, 3.2, 3.3, 4.1, 4.2, 5.1, 7.1, 7.2_

- [x] 4.2 一括 Upsert エンドポイントを定義する
  - PUT `/bulk` エンドポイントを定義し、一括 Upsert 用スキーマでバリデーションする
  - 成功時は 200 ステータスで Upsert 結果の配列を返却する
  - _Requirements: 6.1, 6.2, 6.3, 6.4_

- [x] 4.3 ルートをアプリケーションにマウントする
  - メインの Hono アプリケーションに `/chart-color-settings` パスでルートをマウントする
  - 既存のルート定義の並びに合わせてインポートとマウントを追加する
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 5.1, 6.1_

- [x] 5. テストの実装
- [x] 5.1 (P) API テスト（一覧取得・個別取得・作成・更新・削除）を実装する
  - Hono の `app.request()` を使用してエンドポイントを直接テストする
  - 一覧取得のページネーション動作と `filter[targetType]` フィルタを検証する
  - 個別取得の正常取得と 404 エラーを検証する
  - 作成の 201 レスポンス・`Location` ヘッダー・409 重複エラー・422 バリデーションエラーを検証する
  - 更新の部分更新動作・404 エラー・409 重複エラーを検証する
  - 物理削除の 204 レスポンスと 404 エラーを検証する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 2.1, 2.2, 3.1, 3.2, 3.3, 3.4, 3.5, 4.1, 4.2, 4.3, 4.4, 4.5, 5.1, 5.2_

- [x] 5.2 (P) API テスト（一括 Upsert）を実装する
  - 一括 Upsert の正常動作（新規作成 + 既存更新の混在）を検証する
  - バリデーションエラー時の全体ロールバックを検証する
  - 空配列の送信時の動作を検証する
  - _Requirements: 6.1, 6.2, 6.3, 6.4_

- [x] 5.3 (P) バリデーションスキーマのユニットテストを実装する
  - `colorCode` の有効・無効パターン（`#FF5733` 成功、`FF5733`/`#GGG` 失敗等）を検証する
  - `targetType` の有効・無効値を検証する
  - 更新スキーマで全フィールドオプショナルであることを検証する
  - 一括 Upsert スキーマの配列バリデーションを検証する
  - _Requirements: 7.1, 7.2, 7.3, 7.4_
