# Implementation Plan

- [x] 1. 型定義とバリデーションスキーマの実装
- [x] 1.1 (P) 標準工数マスタの Zod スキーマと TypeScript 型を定義する
  - 重み要素のスキーマ（進捗率: 0〜100 の整数、重み: 0 以上の整数）を定義する
  - 作成用スキーマ（ビジネスユニットコード・案件タイプコード・名称・重み配列）を定義する
  - 更新用スキーマ（名称と重み配列をいずれも任意）を定義する
  - 一覧取得用のクエリパラメータスキーマ（ページネーション・BU/PTフィルタ・論理削除フィルタ）を既存のページネーションスキーマを拡張して定義する
  - DB 行型（マスタ・重み）とAPI レスポンス型（一覧用サマリ・詳細・重み）を定義する
  - ビジネスユニットコード・案件タイプコードには半角英数字とハイフン・アンダースコアの正規表現制約を適用する
  - _Requirements: 7.4, 7.5, 7.6, 8.1, 8.2, 8.3, 8.4, 8.5_

- [x] 1.2 (P) 変換関数（Transform）を実装する
  - DB 行から一覧用レスポンス（サマリ）への変換関数を実装する（snake_case → camelCase、日時を ISO 8601 文字列に変換）
  - DB 行と重みレコードから詳細レスポンスへの変換関数を実装する（重み配列を含む）
  - 重み単体の変換関数を実装する
  - _Requirements: 7.4, 7.5, 7.6_

- [x] 2. データアクセス層の実装
- [x] 2.1 一覧取得・単一取得のデータアクセスを実装する
  - ページネーション付きの一覧取得（総件数と結果セット）を実装する
  - BU コード・PT コードによる動的フィルタリングに対応する
  - 論理削除フィルタ（デフォルト除外、includeDisabled で包含）を実装する
  - ID 指定の単一取得（論理削除済みを除外するもの・含むものの2種）を実装する
  - 複合ユニークキー（BU コード + PT コード + 名称）による検索を実装する
  - マスタ ID に紐づく重みレコードの取得（進捗率の昇順ソート）を実装する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6, 2.1, 2.2, 2.3_

- [x] 2.2 作成・更新のデータアクセスをトランザクション付きで実装する
  - マスタと重みレコードの同時作成をトランザクション内で実行する機能を実装する
  - マスタ更新と重みの全置換（既存削除→新規挿入）をトランザクション内で実行する機能を実装する
  - 名称のみの更新時はトランザクションなしの単純な UPDATE で対応する
  - _Requirements: 3.1, 3.7, 4.1, 4.3_

- [x] 2.3 論理削除・復元・参照チェックのデータアクセスを実装する
  - 論理削除（deleted_at に現在日時を設定）と復元（deleted_at を NULL に、updated_at を更新）を OUTPUT 句で直接結果を返す形式で実装する
  - project_cases テーブルに対する参照存在チェック（論理削除されていないレコードのみ）を実装する
  - _Requirements: 5.1, 5.2, 5.4, 6.1, 6.2_

- [x] 3. サービス層の実装
- [x] 3.1 一覧取得・単一取得のビジネスロジックを実装する
  - データ層の一覧取得結果をサマリレスポンスに変換する処理を実装する
  - 単一取得ではマスタと重みレコードを別々に取得し、詳細レスポンスに結合する処理を実装する
  - マスタが見つからない・論理削除済みの場合に 404 エラーを返す処理を実装する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6, 2.1, 2.2, 2.3_

- [x] 3.2 作成のビジネスロジックを実装する
  - ビジネスユニットコードの存在チェック（存在しなければ 422 エラー）を実装する
  - 案件タイプコードの存在チェック（存在しなければ 422 エラー）を実装する
  - 複合ユニークキーの重複チェック（重複あれば 409 エラー）を実装する
  - データ層での作成後、重みレコードを含む詳細レスポンスを組み立てて返す
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 3.5, 3.6, 3.7, 3.8_

- [x] 3.3 更新のビジネスロジックを実装する
  - 更新対象マスタの存在チェック（見つからなければ 404 エラー）を実装する
  - 名称変更時に同一 BU+PT 内の他レコードとのユニーク制約チェック（自身を除外、重複あれば 409 エラー）を実装する
  - データ層での更新後、重みレコードを含む詳細レスポンスを組み立てて返す
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5, 4.6_

- [x] 3.4 論理削除のビジネスロジックを実装する
  - 削除対象マスタの存在チェック（見つからなければ 404 エラー）を実装する
  - 他リソース（project_cases 等）からの参照チェック（参照あれば 409 エラー）を実装する
  - データ層で論理削除を実行する
  - _Requirements: 5.1, 5.2, 5.3, 5.4_

- [x] 3.5 復元のビジネスロジックを実装する
  - 論理削除済みマスタの存在チェック（見つからない・削除されていなければ 404 エラー）を実装する
  - 復元後の複合ユニークキー重複チェック（重複あれば 409 エラー）を実装する
  - データ層で復元を実行し、重みレコードを含む詳細レスポンスを組み立てて返す
  - _Requirements: 6.1, 6.2, 6.3, 6.4_

- [x] 4. ルート層の実装とシステム統合
- [x] 4.1 各 CRUD エンドポイントのルートハンドラを実装する
  - GET / — 一覧取得（クエリパラメータバリデーション → サービス層呼び出し → data + meta.pagination レスポンス）
  - GET /:id — 単一取得（パスパラメータから ID 取得 → サービス層呼び出し → data レスポンス）
  - POST / — 作成（ボディバリデーション → サービス層呼び出し → 201 + Location ヘッダ + data レスポンス）
  - PUT /:id — 更新（ボディバリデーション → サービス層呼び出し → data レスポンス）
  - DELETE /:id — 論理削除（サービス層呼び出し → 204 No Content）
  - POST /:id/actions/restore — 復元（サービス層呼び出し → data レスポンス）
  - _Requirements: 1.1, 2.1, 3.1, 3.2, 4.1, 5.1, 6.1, 7.1, 7.2, 7.3, 8.1, 8.2, 8.3, 8.4, 8.5_

- [x] 4.2 ルートをアプリケーションにマウントし、エンドツーエンドの動作を確認する
  - `/standard-effort-masters` パスでメインルーターにマウントする
  - ルート型をエクスポートし、型推論チェーンを確保する
  - _Requirements: 7.1, 7.2, 7.3_

  - 重み単体の変換を検証する
  - _Requirements: 9.1_

- [x] 5.3 データアクセス層のテストを実装する
  - 一覧取得の SQL 実行とページネーション、動的 WHERE 句（BU/PT フィルタ）を検証する
  - 単一取得・複合ユニークキー検索・重み取得のパラメータ化クエリを検証する
