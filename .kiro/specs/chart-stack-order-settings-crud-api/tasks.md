## Tasks

- [x] 1. Zodスキーマ・型定義・Transform関数を定義する
- [x] 1.1 (P) 積み上げ順序設定のバリデーションスキーマ、DB行型、APIレスポンス型、変換関数を定義する
  - 作成・更新・一括upsert・一覧クエリの各操作に対応するバリデーションスキーマを定義する
  - `targetType`（必須・文字列・最大20文字）、`targetId`（必須・正の整数）、`stackOrder`（必須・整数）のバリデーションルールを設定する
  - 更新スキーマは全フィールドをオプショナルとする
  - 一括upsertスキーマは `items` 配列（最低1件）を受け付ける構造とする
  - 一覧クエリスキーマはページネーションパラメータ（デフォルト値: page=1, pageSize=20）と `targetType` フィルタを定義する
  - DB行型（snake_case）とAPIレスポンス型（camelCase）を定義する
  - DB行からAPIレスポンスへの変換関数を実装し、日付型をISO 8601文字列に変換する
  - _Requirements: 1.6, 7.1, 7.2, 7.3, 7.4, 7.5_

- [x] 2. データアクセス層を実装する
- [x] 2.1 (P) 標準CRUD操作のデータアクセス関数を実装する
  - ページネーション付き一覧取得（件数取得と結果取得を分離実行）を実装する
  - `targetType` による動的フィルタリングを実装する
  - IDによる個別取得を実装する
  - `(targetType, targetId)` 複合キーの存在チェック関数を実装する（自身除外オプション付き）
  - 新規作成（INSERT + 作成行の取得）を実装する
  - 更新（UPDATE + 更新行の取得）を実装する
  - 物理削除を実装する
  - すべてのクエリでパラメータ化SQLを使用し、カラムリスト定数を一元管理する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 2.1, 3.1, 4.1, 5.1_

- [x] 2.2 (P) 一括upsertのデータアクセス関数を実装する
  - トランザクション内で `MERGE` 文を使用し、`(targetType, targetId)` をマッチキーとしたupsert操作を実装する
  - 一部でも失敗した場合は全体をロールバックする
  - upsert完了後に全対象行を取得して返却する
  - _Requirements: 6.1, 6.4_

- [x] 3. サービス層を実装する
- [x] 3.1 標準CRUD操作のビジネスロジックを実装する
  - ページネーション付き一覧取得（Transformを介したレスポンス変換とページネーションメタ情報の生成）を実装する
  - IDによる個別取得（存在しない場合は404エラー）を実装する
  - 新規作成（`(targetType, targetId)` の重複チェックで409エラー、正常時はTransform済みレスポンス返却）を実装する
  - 更新（存在確認で404エラー、重複チェックで409エラー、自身を除外した複合キーチェック）を実装する
  - 物理削除（存在確認で404エラー）を実装する
  - すべての正常レスポンスはTransform経由でDB行型からAPI型に変換する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6, 2.1, 2.2, 3.1, 3.2, 3.3, 3.4, 4.1, 4.2, 4.3, 4.4, 5.1, 5.2, 8.1, 8.2_

- [x] 3.2 一括upsertのビジネスロジックを実装する
  - リクエスト内の `items` 配列で `(targetType, targetId)` の入力重複を検証し、重複がある場合は422エラーを返却する
  - データ層の一括upsert関数を呼び出し、結果をTransform経由で変換する
  - _Requirements: 6.1, 6.2, 6.3, 8.1_

- [x] 4. ルート層を実装しシステムに統合する
- [x] 4.1 全エンドポイントのルートハンドラを定義し、アプリケーションに登録する
  - GET `/` — `validate` ミドルウェアでクエリパラメータを検証し、一覧取得サービスを呼び出してページネーション付きレスポンスを返却する
  - GET `/:id` — `parseIntParam` でパスパラメータを検証し、個別取得サービスを呼び出す
  - POST `/` — `validate` でリクエストボディを検証し、作成サービスを呼び出し、201ステータスと `Location` ヘッダーを設定する
  - PUT `/bulk` — `validate` でリクエストボディを検証し、一括upsertサービスを呼び出す（`/:id` より前にルート定義すること）
  - PUT `/:id` — `parseIntParam` + `validate` で検証し、更新サービスを呼び出す
  - DELETE `/:id` — `parseIntParam` で検証し、削除サービスを呼び出して204空レスポンスを返却する
  - アプリケーションのエントリポイントにルートを登録する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6, 2.1, 2.2, 3.1, 3.2, 3.3, 3.4, 4.1, 4.2, 4.3, 4.4, 5.1, 5.2, 6.1, 6.2, 6.3, 6.4, 7.5, 8.1, 8.2, 8.3_
