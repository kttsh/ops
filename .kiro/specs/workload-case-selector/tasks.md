# Implementation Plan

- [x] 1. Backend - プロジェクト一覧 API にケースサマリを追加
- [x] 1.1 (P) プロジェクト一覧のレスポンスに各案件のケース概要を含める
  - プロジェクト一覧取得時に、各案件に紐づくケース（project_cases テーブル）の概要情報（ケースID、ケース名、isPrimary）を追加で取得する
  - メインの案件取得クエリとは別クエリでケースサマリを一括取得し、サービス層で案件IDをキーにして結合する
  - ソフトデリート済みケース（deleted_at IS NOT NULL）は除外する
  - Transform 層でケースサマリを camelCase の API レスポンス形式に変換する
  - バックエンドのプロジェクトレスポンス型に cases フィールドを追加する
  - _Requirements: 1.2_

- [x] 2. Backend - chart-data API のケースID指定対応
- [x] 2.1 (P) chart-data API にケースIDパラメータのバリデーションとパラメータ受け渡しを追加
  - chart-data のクエリパラメータにカンマ区切りのケースID配列（projectCaseIds）を受け付けるバリデーションを追加する
  - サービス層のパラメータ型にケースID配列を追加する
  - ルート定義からサービス呼び出しへの projectCaseIds パラメータ受け渡しを追加する
  - _Requirements: 4.1_

- [x] 2.2 ケースオーバーライド対応のデータ取得メソッドとサービス分岐を実装
  - 指定されたケースIDのデータを取得しつつ、未指定の案件は isPrimary ケースにフォールバックする SQL クエリを新規メソッドとして実装する
  - 既存の chartView パターン（指定ケースID優先、未指定は isPrimary フォールバック）の条件分岐を参考にする
  - サービス層で chartViewId → projectCaseIds → デフォルト の3段階分岐を実装する
  - projectIds と projectCaseIds の両方が指定された場合、projectIds が案件スコープを決定し、projectCaseIds はスコープ内のケースオーバーライドとして機能する
  - 不正なケースID（存在しない、または指定案件に属さない）は無視し、該当案件は isPrimary にフォールバックする
  - _Requirements: 4.2, 4.3, 4.4, 3.2_

- [x] 3. Frontend - 型定義と API クライアントの拡張
- [x] 3.1 (P) フロントエンドの型定義と API クライアントをケース対応に拡張
  - チャートデータのパラメータ型にケースID配列フィールドを追加する
  - プロジェクト型にケースサマリ（ID、ケース名、isPrimary）のリストを追加する
  - チャートデータ取得関数で、ケースIDパラメータが指定されている場合のみ HTTP リクエストのクエリパラメータに含めるようにする
  - _Requirements: 1.2, 3.1_

- [x] 4. Frontend - ケース選択状態管理とパラメータ構築
- [x] 4.1 ワークロードページにケース選択状態の管理ロジックを実装
  - ワークロードページに案件IDからケースIDへのマッピング状態を追加する
  - 案件データ取得後に各案件の isPrimary ケースをデフォルト選択する初期化ロジックを実装する
  - isPrimary ケースが存在しない案件は最初のケースをデフォルトとして選択する
  - 案件チェック変更時に、新規チェック案件のデフォルトケース自動選択とチェック解除案件のケース状態クリアを行う
  - 再チェック時はデフォルトケース（isPrimary）で再選択する
  - 他の案件のケースを変更しても、既に選択済みの案件のケース選択状態は維持する
  - _Requirements: 2.1, 2.2, 2.3, 5.1, 5.2, 5.3_

- [x] 4.2 チャートデータパラメータへのケースID反映と省略最適化を実装
  - チャートデータのパラメータ構築ロジックに選択中のケースIDの配列を追加する
  - 全案件が isPrimary ケースを選択している場合（全デフォルト状態）はケースIDパラメータを省略し、既存の最適化クエリパスを使用する
  - 1件でも非デフォルトケースが選択されている場合のみケースIDパラメータを送信する
  - _Requirements: 3.1, 3.2_

- [x] 5. Frontend - ケースセレクタ UI の実装
- [x] 5.1 サイドパネルの案件カードにケースセレクタを追加
  - サイドパネルの案件カードコンポーネントの props にケース選択状態とケース変更コールバックを追加する
  - 案件カード内にケース選択ドロップダウンを配置し、各案件に紐づく全ケースを選択肢として表示する
  - 各ケースのケース名を選択肢のラベルとして使用する
  - 案件に紐づくケースが1件のみの場合はセレクタを非表示にする
  - ケース変更時に親コンポーネント（ワークロードページ）のケース選択状態を更新し、チャートデータの再取得をトリガーする
  - ワークロードページからケースセレクタへの props 受け渡しを接続する
  - データ取得中はローディング状態が既存の仕組みで表示される
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 3.1, 3.3_

- [x] 6. テスト
- [x] 6.1 (P) バックエンドのテストを実装
  - chart-data API に projectCaseIds を指定した場合に指定ケースのデータが返却されることを検証する
  - projectCaseIds 未指定時に既存動作（isPrimary ケース）が維持されることを検証する（後方互換テスト）
  - 不正な projectCaseIds 指定時に isPrimary へフォールバックすることを検証する
  - プロジェクト一覧 API のレスポンスにケースサマリが含まれることを検証する
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 1.2_

- [x] 6.2 (P) フロントエンドのテストを実装
  - デフォルトケース初期化ロジックのテスト（isPrimary 選択、isPrimary なしのフォールバック、空ケースリスト）
  - チャートデータパラメータ構築ロジックのテスト（全デフォルト時の省略、カスタム選択時の含有）
  - _Requirements: 2.1, 2.3, 3.1_
