# Implementation Plan

- [x] 1. DB スキーマ変更とバックエンド型定義の基盤整備
- [x] 1.1 projects テーブルへの 7 カラム追加
  - `create-tables.sql` に 7 つの NULL 許容カラム（fiscal_year, nickname, customer_name, order_number, calculation_basis, remarks, region）を追加
  - ALTER TABLE 文を作成し、DB に適用可能な状態にする
  - テーブル仕様書（`docs/database/table-spec.md`）の projects セクションに 7 カラムの定義を追記
  - _Requirements: 1.4_

- [x] 1.2 (P) バックエンドプロジェクト型定義と Transform の拡張
  - プロジェクトの作成・更新スキーマに 7 つの新規フィールドを追加（全て任意・NULL 許容）
  - DB 行型と API レスポンス型に 7 つの新規フィールドを追加
  - プロジェクト Transform にフィールドマッピングを追加
  - プロジェクトデータ層の SELECT カラム一覧、INSERT 文、UPDATE の fieldMap に 7 カラムを反映
  - _Requirements: 1.4_

- [x] 1.3 (P) インポート・エクスポート型定義の刷新
  - エクスポート行型を 17 固定列（案件メタデータ + ケース情報）に拡張
  - エクスポートレスポンス型を更新
  - インポートアイテム型を 17 固定列 + 月次データ配列に再定義
  - インポートリクエストスキーマを新しいアイテム型に基づいて Zod で定義
  - インポート結果型に createdProjects, updatedProjects, deletedProjects, createdCases を追加
  - DB エクスポート行型に案件メタデータカラムを追加
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 3.1, 4.1, 7.1_

- [x] 2. バックエンドデータ層の拡張
- [x] 2.1 (P) エクスポート SQL の拡張
  - エクスポートクエリに projects テーブルの全メタデータカラム（既存 + 新規 7 カラム）を追加
  - ソート順を案件コード昇順 → ケース ID 昇順に変更
  - DB エクスポート行型を新カラムに合わせて更新
  - _Requirements: 5.1, 5.2, 5.3_

- [x] 2.2 (P) マスタ存在チェックのバッチ検証メソッド追加
  - ビジネスユニットコードのバッチ検証メソッドを追加（入力コード一覧から無効なコードを返却）
  - プロジェクトタイプコードのバッチ検証メソッドを追加（同上）
  - 既存の `validateProjectCaseIds` パターン（IN 句 + パラメータ化クエリ）を踏襲
  - _Requirements: 2.2, 2.4, 9.2, 9.3_

- [x] 2.3 案件コード自動生成と案件 CRUD メソッド追加
  - 次の AUTO 案件コード（`AUTO-{000001}` 形式）を生成するメソッドを追加。トランザクション内で `LIKE 'AUTO-%'` の MAX を取得し +1
  - 案件コードで既存案件をバッチ検索するメソッドを追加
  - トランザクション内で案件を INSERT するメソッドを追加（7 新カラム含む全フィールド）
  - トランザクション内で案件メタデータを UPDATE するメソッドを追加
  - トランザクション内で案件をソフトデリートするメソッドを追加
  - _Requirements: 3.1, 3.2, 3.3, 7.1, 7.2_

- [x] 2.4 ケース CRUD と月次データ MERGE メソッド追加
  - トランザクション内でケースを INSERT するメソッドを追加（projectId + caseName）
  - 既存の MERGE パターンを維持した月次データ upsert メソッドを追加（トランザクション対応）
  - 既存の `bulkImportByCase` は新しいオーケストレーションに置き換えるため削除対象
  - _Requirements: 4.1, 4.2, 4.3_

- [x] 3. バックエンドサービス・ルートの刷新
- [x] 3.1 エクスポートサービスの刷新
  - `getExportData()` を案件メタデータ付きエクスポート行型でケース単位にグループ化するよう更新
  - 年月範囲生成ロジック（`generateYearMonths`）は既存を維持
  - データ空時の空レスポンス返却を維持
  - _Requirements: 5.1, 5.2, 5.3, 5.5, 8.3_

- [x] 3.2 インポートサービスのオーケストレーション刷新
  - 入力データから一意な BU コード・PT コードを抽出し、マスタ存在チェックを実行（422 エラー）
  - トランザクションを開始し、以下を順次実行:
    - 案件コードで既存案件をバッチ検索
    - 案件コード空欄行に AUTO コードを生成して案件 INSERT
    - 案件コード既存行の案件メタデータを UPDATE
    - ケース ID 空欄行に新規ケースを INSERT
    - 各行の月次データを MERGE
    - O 列が削除指定の行に対してソフトデリート実行
  - トランザクションをコミットし、処理件数を BulkImportResult として返却
  - エラー時はトランザクションをロールバック
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 3.1, 3.2, 3.3, 3.4, 4.1, 4.2, 4.3, 7.1, 7.2, 9.1_

- [x] 3.3 ルートの新スキーマバリデーション適用
  - POST エンドポイントの Zod バリデーションを新しい `bulkImportSchema` に切り替え
  - GET エンドポイントのレスポンス型を新しい `ExportDataResponse` に更新
  - エンドポイントパス（`/export-project-loads`, `/import-project-loads`）は変更なし
  - _Requirements: 9.1_

- [x] 4. フロントエンドの刷新
- [x] 4.1 (P) Excel ユーティリティへの YY-MM 変換関数追加
  - `convertShortYearMonthHeader` 関数を追加: YY-MM → YYYYMM（例: "25-04" → "202504"）
  - `formatShortYearMonth` 関数を追加: YYYYMM → YY-MM（例: "202504" → "25-04"）
  - 20 世紀/21 世紀の切り替えルール: YY >= 70 → 19YY、YY < 70 → 20YY
  - 既存の `convertYearMonthHeader`（YYYY-MM 形式用）は他機能で利用中のため維持
  - _Requirements: 1.2, 1.3, 6.2, 6.3_

- [x] 4.2 (P) Bulk API クライアントの型定義更新
  - エクスポート行型を 17 固定列対応に拡張（案件メタデータ全フィールド）
  - インポートリクエスト型を BulkImportRow 配列に変更
  - インポート結果型に createdProjects, deletedProjects, createdCases を追加
  - エンドポイント URL は変更なし
  - _Requirements: 1.1, 5.1_

- [x] 4.3 エクスポート hook の 17 固定列対応
  - fixedHeaders を 17 列に拡張: 案件コード, BU, 年度, 工事種別, 案件名, 通称・略称, 客先名, オーダ, 開始時期, 案件工数, 月数, 算出根拠, 備考, 地域, 削除, 工数ケースNo, 工数ケース名
  - 年月列ヘッダーを YY-MM 形式で出力（`formatShortYearMonth` を使用）
  - fixedValues にエクスポート行の全メタデータフィールドをマッピング
  - データ空時の警告トースト表示を維持
  - ローディング状態管理と二重クリック防止を維持
  - _Requirements: 1.1, 1.2, 1.3, 5.1, 5.2, 5.3, 5.4, 5.5, 7.3, 8.1, 8.2, 8.3_

- [x] 4.4 インポート hook の 17 固定列パース・バリデーション・確定処理
  - `fixedColumnCount: 17` で `parseBulkImportSheet` を呼び出し
  - `parseYearMonth` に `convertShortYearMonthHeader` を使用（YY-MM → YYYYMM）
  - `validateRow` で以下のバリデーションを実施:
    - B列（BU）: 必須チェック
    - E列（案件名）: 案件コード空欄時に必須チェック
    - I列（開始時期）: YYYYMM 形式チェック、案件コード空欄時に必須
    - J列（全体工数）: 数値・範囲チェック（0〜99,999,999）、案件コード空欄時に必須
    - Q列（ケース名）: ケースNo空欄時に必須チェック
  - R列以降の工数値バリデーション（0〜99,999,999）
  - プレビュー表示用 label を `{projectCode} | {name} / {caseName}` 形式で生成
  - 確定処理で 17 固定列データを BulkImportRow 配列に変換し API 送信
  - 成功時にトースト表示 + クエリ無効化で画面更新
  - エラー時にエラートースト表示
  - _Requirements: 1.1, 2.1, 2.2, 2.3, 2.4, 3.1, 3.4, 4.1, 4.3, 6.1, 6.2, 6.3, 6.4, 7.1, 7.2, 9.1, 9.2, 9.3, 9.4, 9.5, 9.6, 9.7, 9.8, 9.9_
