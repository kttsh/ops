# Requirements Document

## Introduction

標準工数マスタのメンテナンス画面（一覧・新規作成・詳細/編集）をフロントエンドに実装する。
バックエンド API は実装済みであり、フロントエンドのルート・features・サイドバーメニューが未実装の状態である。

標準工数マスタは、案件の進捗率（0-100%、5%刻み・21区間）ごとの工数配分の重み（weight）を定義するテンプレートである。均等配分・Sカーブ・バスタブカーブなどの工数カーブを表現し、計画案件の月別工数按分に使用される。

既存マスタ画面（ビジネスユニット / 案件タイプ / 作業種類 / 案件）と同様のページ型パターンを踏襲し、一貫した操作体験を提供する。

## Requirements

### Requirement 1: 一覧画面

**Objective:** マスタ管理者として、登録済みの標準工数パターンを一覧で確認・検索・フィルタリングしたい。パターンの全体像を素早く把握し、目的のレコードに効率的にアクセスするためである。

#### Acceptance Criteria

1. When ユーザーが `/master/standard-effort-masters/` に遷移した時, the 一覧画面 shall DataTable 形式で標準工数マスタの一覧を表示する（カラム: パターン名 / ビジネスユニット / 案件タイプ / ステータス / 更新日時）
2. When ユーザーが検索テキストを入力した時, the 一覧画面 shall パターン名でフィルタリングされた結果を表示する
3. When ユーザーがビジネスユニットまたは案件タイプのフィルタを選択した時, the 一覧画面 shall 選択された条件に合致するレコードのみを表示する
4. When ユーザーが「無効データ表示」トグルを切り替えた時, the 一覧画面 shall 論理削除済みレコードの表示/非表示を切り替える
5. The 一覧画面 shall ページネーションによるページ分割表示をサポートする
6. When ユーザーがデータ行をクリックした時, the 一覧画面 shall 該当レコードの詳細/編集画面に遷移する
7. When ユーザーが「新規作成」ボタンをクリックした時, the 一覧画面 shall 新規作成画面に遷移する

### Requirement 2: 新規作成画面

**Objective:** マスタ管理者として、新しい標準工数パターンを登録したい。BU・案件タイプごとに工数カーブの重み配分を定義し、計画案件に適用できるようにするためである。

#### Acceptance Criteria

1. When ユーザーが `/master/standard-effort-masters/new` に遷移した時, the 新規作成画面 shall ビジネスユニット（QuerySelect）、案件タイプ（QuerySelect）、パターン名（テキスト入力）、および21行の重み入力テーブル（進捗率 0%-100%、5%刻み）を含むフォームを表示する
2. While ユーザーが重み値を入力している間, the 新規作成画面 shall 入力値に基づくエリアチャート（横軸: 進捗率、縦軸: 重み）をリアルタイムで更新・表示する
3. When ユーザーが有効な値を入力してフォームを送信した時, the 新規作成画面 shall バックエンド API に POST リクエストを送信し、成功時にトースト通知を表示して一覧画面に遷移する
4. If 必須フィールドが未入力またはバリデーションエラーが存在する場合, the 新規作成画面 shall 該当フィールドにエラーメッセージを表示し、送信を抑止する
5. If API がエラーレスポンスを返した場合, the 新規作成画面 shall エラー内容をユーザーに通知する

### Requirement 3: 詳細/編集画面

**Objective:** マスタ管理者として、既存の標準工数パターンの詳細を確認し、必要に応じて編集・削除・復元したい。パターンの内容を正確に把握し、適切にメンテナンスするためである。

#### Acceptance Criteria

1. When ユーザーが `/master/standard-effort-masters/$standardEffortId/` に遷移した時, the 詳細/編集画面 shall 閲覧モードでマスタ情報（パターン名、ビジネスユニット、案件タイプ）、重み値テーブル（読み取り専用）、およびエリアチャートを表示する
2. When ユーザーが「編集」ボタンをクリックした時, the 詳細/編集画面 shall 編集モードに切り替え、パターン名と重み値を編集可能にする（ビジネスユニット・案件タイプは変更不可）
3. While 編集モードでユーザーが重み値を変更している間, the 詳細/編集画面 shall エリアチャートをリアルタイムで更新する
4. When ユーザーが編集モードで「保存」ボタンをクリックした時, the 詳細/編集画面 shall バックエンド API に PUT リクエストを送信し、成功時にトースト通知を表示して閲覧モードに戻る
5. When ユーザーが編集モードで「キャンセル」ボタンをクリックした時, the 詳細/編集画面 shall 変更を破棄して閲覧モードに戻る
6. When ユーザーが「削除」ボタンをクリックした時, the 詳細/編集画面 shall 削除確認ダイアログを表示する
7. When ユーザーが削除確認ダイアログで確認した時, the 詳細/編集画面 shall バックエンド API に DELETE リクエストを送信し、成功時にトースト通知を表示して一覧画面に遷移する
8. When 論理削除済みレコードの詳細画面でユーザーが「復元」ボタンをクリックした時, the 詳細/編集画面 shall 復元確認ダイアログを表示し、確認後に復元 API を呼び出す
9. If API がエラーレスポンスを返した場合, the 詳細/編集画面 shall エラー内容をユーザーに通知する

### Requirement 4: エリアチャート（重み分布可視化）

**Objective:** マスタ管理者として、入力した重み値の分布をチャートで視覚的に確認したい。工数カーブの形状（均等・Sカーブ・バスタブカーブ等）を直感的に把握するためである。

#### Acceptance Criteria

1. The エリアチャート shall Recharts AreaChart を使用し、横軸に進捗率（0%-100%）、縦軸に重みを表示する
2. The エリアチャート shall 新規作成画面と詳細/編集画面の両方で再利用可能なコンポーネントとして実装される
3. While フォームの重み値が変更されている間, the エリアチャート shall リアルタイムに描画を更新する

### Requirement 5: ナビゲーション

**Objective:** ユーザーとして、サイドバーメニューから標準工数マスタ画面にアクセスしたい。他のマスタ画面と同じ導線で一貫した操作体験を得るためである。

#### Acceptance Criteria

1. The サイドバーナビゲーション shall 「標準工数パターン」メニュー項目をマスタ管理セクションに表示する
2. When ユーザーが「標準工数パターン」メニュー項目をクリックした時, the サイドバーナビゲーション shall 標準工数マスタの一覧画面に遷移する

### Requirement 6: 既存パターンとの一貫性

**Objective:** 開発者として、既存マスタ画面のコーディングパターンに準拠した実装を行いたい。コードベースの一貫性を保ち、保守性を確保するためである。

#### Acceptance Criteria

1. The 標準工数マスタ機能 shall 既存マスタ画面で使用されている共通コンポーネント（DataTable / PageHeader / FormTextField / QuerySelect / DeleteConfirmDialog / RestoreConfirmDialog / StatusBadge 等）を再利用する
2. The 標準工数マスタ機能 shall 既存の API クライアントパターン（api-client / queries / mutations / createQueryKeys / createListQueryOptions / createDetailQueryOptions / createCrudMutations）に準拠する
3. The 標準工数マスタ機能 shall 既存のカラム定義ヘルパー（createSortableColumn / createStatusColumn 等）を使用する
4. The 標準工数マスタ機能 shall TypeScript エラーなしでコンパイルされる
5. The 標準工数マスタ機能 shall 既存テストを破壊しない
