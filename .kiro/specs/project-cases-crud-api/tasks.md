# Implementation Plan

- [x] 1. 型定義とバリデーションスキーマの作成
- [x] 1.1 案件ケースの Zod バリデーションスキーマと TypeScript 型を定義する
  - 作成用スキーマを定義する: ケース名（必須・最大100文字）、プライマリフラグ（デフォルト false）、説明（任意・最大500文字）、計算タイプ（MANUAL/STANDARD、デフォルト MANUAL）、標準工数パターンID（任意・正の整数）、開始年月（任意・YYYYMM形式6桁）、期間月数（任意・正の整数）、総工数（任意・非負整数）
  - 更新用スキーマを定義する: 作成用と同一フィールドだが全て任意。NULL を許容するフィールド（description, standardEffortId, startYearMonth, durationMonths, totalManhour）は nullable とする
  - 一覧取得クエリスキーマを定義する: 既存のページネーションスキーマを拡張し、ソフトデリート済みレコード包含フィルタを追加する
  - DB 行型を定義する: snake_case のカラム名に加えて、JOIN で取得する案件名と標準工数パターン名のフィールドを含める
  - API レスポンス型を定義する: camelCase に変換した全フィールド（JOIN 名称含む）を持つ型を定義する
  - _Requirements: 3.3, 4.2, 7.2, 7.5_
  - _Contracts: projectCase types Service Interface_

- [x] 2. データ変換層の作成
- [x] 2.1 (P) DB 行データから API レスポンス形式への変換関数を作成する
  - DB の snake_case フィールドを camelCase の API レスポンスフィールドに変換する関数を作成する
  - 日時フィールド（created_at, updated_at）を ISO 8601 文字列に変換する
  - JOIN で取得した案件名と標準工数パターン名もレスポンスに含める
  - 論理削除日時（deleted_at）はレスポンスに含めない
  - _Requirements: 1.5, 2.2, 7.4_
  - _Contracts: projectCaseTransform Service Interface_

- [x] 3. データアクセス層の作成
- [x] 3.1 一覧取得と単一取得のクエリを実装する
  - 指定された案件IDに紐づくケース一覧を取得するクエリを作成する。案件テーブルと標準工数パターンテーブルを JOIN して名称を取得する
  - ページネーション対応（OFFSET/FETCH）と件数カウントクエリを実装する
  - ソフトデリート済みレコードの除外/包含をフィルタパラメータで制御する
  - 一覧の並び順は作成日時の昇順とする
  - 単一取得クエリを作成する。JOIN で名称を含め、ソフトデリート済みを除外する
  - 復元用に、ソフトデリート済みを含む単一取得クエリも作成する
  - _Requirements: 1.1, 1.3, 1.4, 1.5, 1.7, 2.1, 2.2_
  - _Contracts: projectCaseData findAll, findById, findByIdIncludingDeleted_

- [x] 3.2 作成・更新・論理削除・復元のクエリを実装する
  - 新規レコード挿入クエリを作成する。OUTPUT 句で自動採番IDを取得後、JOIN 付きの単一取得で完全な行を返す
  - 更新クエリを作成する。指定されたフィールドのみ動的に SET 句を構成し、updated_at を現在日時に更新する。ソフトデリート済みは更新対象外とする
  - 論理削除クエリを作成する。deleted_at と updated_at を現在日時に設定する
  - 復元クエリを作成する。deleted_at を NULL に戻し updated_at を現在日時に更新する
  - _Requirements: 3.1, 4.1, 4.5, 5.1, 6.1_
  - _Contracts: projectCaseData create, update, softDelete, restore_

- [x] 3.3 参照チェックと存在確認のクエリを実装する
  - 指定された案件ケースが月次負荷データやチャートビュー項目から参照されているかを EXISTS で確認するクエリを作成する
  - 案件IDの存在確認クエリを作成する（親リソースチェック用）
  - 標準工数パターンIDの存在確認クエリを作成する（作成・更新時のバリデーション用）
  - _Requirements: 5.3, 3.5, 3.7, 1.6_
  - _Contracts: projectCaseData hasReferences, projectExists, standardEffortExists_

- [x] 4. サービス層の作成
- [x] 4.1 一覧取得と単一取得のビジネスロジックを実装する
  - 一覧取得: 指定された案件が存在するか確認し、存在しない場合は 404 エラーを送出する。データ層から取得した結果を変換してページネーション情報とともに返却する
  - 単一取得: 指定された案件ケースを取得し、存在しない場合またはソフトデリート済みの場合は 404 エラーを送出する。取得した案件ケースの親案件IDがURLの案件IDと一致しない場合も 404 を送出する
  - _Requirements: 1.6, 2.3, 2.4_
  - _Contracts: projectCaseService findAll, findById_

- [x] 4.2 作成のビジネスロジックを実装する
  - 親案件の存在確認を行い、存在しない場合は 404 エラーを送出する
  - 計算タイプが STANDARD の場合に標準工数パターンIDが未指定であれば 422 エラーを送出する
  - 標準工数パターンIDが指定されている場合、そのIDの存在を確認し、存在しない場合は 422 エラーを送出する
  - データ層で作成後、変換層を通じてレスポンスを返却する
  - _Requirements: 3.5, 3.6, 3.7_
  - _Contracts: projectCaseService create_

- [x] 4.3 更新のビジネスロジックを実装する
  - 対象ケースの存在確認を行い、存在しないまたはソフトデリート済みの場合は 404 エラーを送出する
  - 親案件IDの整合性チェックを行う
  - 更新後の計算タイプが STANDARD になる場合（更新データで変更する場合、または既存値が STANDARD で変更しない場合）、標準工数パターンIDが設定されているか確認する。未設定なら 422 エラーを送出する
  - データ層で更新後、変換層を通じてレスポンスを返却する
  - _Requirements: 4.3, 4.6_
  - _Contracts: projectCaseService update_

- [x] 4.4 論理削除と復元のビジネスロジックを実装する
  - 論理削除: 他リソースからの参照チェックを行い、参照されている場合は 409 エラーを送出する。対象ケースが存在しないまたは既にソフトデリート済みの場合は 404 エラーを送出する
  - 復元: 対象ケースが存在しない場合は 404 エラー、ソフトデリートされていない場合は 409 エラーを送出する。復元後のリソースを返却する
  - _Requirements: 5.2, 5.3, 6.2, 6.3_
  - _Contracts: projectCaseService delete, restore_

- [x] 5. ルート層の作成とシステム統合
- [x] 5.1 CRUD エンドポイントを定義する
  - 一覧取得エンドポイントを定義する: クエリパラメータをバリデーションし、サービス層に委譲してページネーション付きレスポンスを返却する
  - 単一取得エンドポイントを定義する: パスパラメータを整数に変換してサービス層に委譲する
  - 作成エンドポイントを定義する: リクエストボディをバリデーションし、サービス層で作成後、Location ヘッダとともに 201 を返却する
  - 更新エンドポイントを定義する: リクエストボディをバリデーションし、サービス層で更新後 200 を返却する
  - 削除エンドポイントを定義する: サービス層で論理削除後 204 No Content を返却する
  - 復元エンドポイントを定義する: サービス層で復元後 200 を返却する
  - パスパラメータ（projectId, projectCaseId）を各ハンドラで整数に変換し、変換不能な場合は 422 エラーを送出する
  - メソッドチェーンで定義し、RPC 型推論用にルート型をエクスポートする
  - _Requirements: 1.1, 1.2, 2.1, 3.1, 3.2, 3.4, 4.1, 4.4, 5.1, 6.1, 7.1, 7.2, 7.3_
  - _Contracts: projectCases Route API Contract_

- [x] 5.2 アプリケーションにルートをマウントする
  - エントリーポイントに案件ケースルートをネストルートとしてマウントする（/projects/:projectId/project-cases の階層）
  - 既存のルートマウントとの整合性を確認する
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 5.1, 6.1_

- [x] 6. テストの作成
- [x] 6.1 一覧取得と単一取得のテストを作成する
  - 一覧取得の正常系: ページネーション付きレスポンス、JOIN 名称の含有、ソフトデリート除外、includeDisabled フィルタの動作を検証する
  - 単一取得の正常系: 詳細レスポンスと JOIN 名称の含有を検証する
  - 一覧取得の異常系: 存在しない案件IDでの 404 を検証する
  - 単一取得の異常系: 存在しないケースID → 404、案件ID不一致 → 404 を検証する
  - _Requirements: 8.1, 8.2_

- [x] 6.2 作成と更新のテストを作成する
  - 作成の正常系: 201 ステータス、Location ヘッダ、レスポンスボディを検証する
  - 更新の正常系: 200 ステータス、更新されたフィールドの反映を検証する
  - 作成の異常系: バリデーションエラー → 422、存在しない案件ID → 404、STANDARD + standardEffortId 未指定 → 422、存在しない standardEffortId → 422 を検証する
  - 更新の異常系: 存在しないケース → 404、バリデーションエラー → 422、STANDARD 変更時の整合性エラー → 422 を検証する
  - _Requirements: 8.1, 8.2_

- [x] 6.3 論理削除と復元のテストを作成する
  - 論理削除の正常系: 204 No Content を検証する
  - 復元の正常系: 200 ステータスと復元されたリソースを検証する
  - 論理削除の異常系: 存在しないケース → 404、参照あり → 409 を検証する
  - 復元の異常系: 存在しないケース → 404、未削除ケースの復元 → 409 を検証する
  - _Requirements: 8.1, 8.2_
  - _Contracts: Vitest + app.request() パターン_
