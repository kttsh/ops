### Issue #1: 一括入力で初期値→到達値の按分入力

**画面**: `/master/headcount-plans`
**カテゴリ**: 機能追加
**報告内容**: 一括入力で初期値と到達値から按分。4月と3月を指定して、中間の月は四捨五入でばらす。

#### 現状の実装

- **一括入力ダイアログ** (`BulkInputDialog.tsx`): 年度と人員数を指定し、全12ヶ月に**同一値**を設定する
- 按分（線形補間）機能は未実装
- 月グリッド: `MonthlyHeadcountGrid.tsx` で4月〜翌3月の12ヶ月入力

#### 要件（案）

| 項目 | 内容 |
|------|------|
| 機能名 | 按分入力（線形補間）|
| 入力パラメータ | 開始月値（4月）、終了月値（3月）|
| 計算ロジック | 線形補間で中間月を算出、四捨五入で整数化 |
| 対象期間 | 選択した年度の12ヶ月（4月〜翌3月）|
| UI | 既存の一括入力ダイアログに「按分モード」を追加（トグル切替）|

#### 計算ロジック例

```
開始値: 10人（4月）、到達値: 15人（3月）
ステップ = (15 - 10) / 11 = 0.4545...

4月: 10, 5月: round(10.45)=10, 6月: round(10.91)=11, 7月: round(11.36)=11,
8月: round(11.82)=12, 9月: round(12.27)=12, 10月: round(12.73)=13,
11月: round(13.18)=13, 12月: round(13.64)=14, 1月: round(14.09)=14,
2月: round(14.55)=15, 3月: 15
```

#### 影響範囲

| ファイル | 変更内容 |
|----------|---------|
| `BulkInputDialog.tsx` | 按分モードUI追加、計算ロジック追加 |
| `useHeadcountPlansPage.ts` | 必要に応じてインターフェース変更 |

#### Open Questions

- [Q1-1] 按分は常に4月→3月の12ヶ月固定か？任意の開始月→終了月を指定可能にすべきか？
- [Q1-2] 既存の「全月同一値」モードと按分モードはどちらをデフォルトにすべきか？

### Issue #6: グラフエリアのArea部クリック時の黒枠

**画面**: `/workload`
**カテゴリ**: バグ修正
**報告内容**: グラフエリアのArea部をクリックすると黒枠が出る

#### 現状の実装

- `WorkloadChart.tsx` (L91): `style={{ outline: "none" }}` が `ComposedChart` に設定済み
- `activeDot={false}` で各 Area/Line のアクティブドットを無効化済み
- **しかしユーザーから黒枠が報告されている**

#### 原因推定

- `outline: "none"` は `ComposedChart` の SVG コンテナに対する設定
- **Area 要素自体** や **SVG `<path>` 要素** がフォーカスを受けた際のブラウザデフォルトスタイル（`outline` or CSS `focus` ring）が原因の可能性
- Recharts が内部で生成する `<path>` や `<rect>` に `tabIndex` が設定されている可能性

#### 修正案

```css
/* globals.css に追加 */
.recharts-surface:focus,
.recharts-surface *:focus {
  outline: none;
}
```

または `WorkloadChart.tsx` で:

```tsx
<ComposedChart
  style={{ outline: "none" }}
  accessibilityLayer={false}  // ← 既にfalse
  // 追加: cursor="pointer" をやめ、SVGのfocusableを無効化
>
```

#### 影響範囲

| ファイル | 変更内容 |
|----------|---------|
| `globals.css` | Recharts SVG要素のフォーカス枠を抑制 |
| `WorkloadChart.tsx` | 必要に応じて追加のスタイル設定 |

- `POST /capacity-scenarios/:id/actions/calculate` で自動計算API実装済み
  - ただし1つのシナリオ × 1つの人員計画ケースの組み合わせで計算
- ダッシュボードでは `capacityScenarioIds` を指定してキャパシティラインを表示

#### 要件

| 項目 | 内容 |
|------|------|
| 計算 | 人員計画ケース(n個) × キャパシティシナリオ(m個) = n×m本のキャパシティライン |

#### 影響範囲

**方式A: フロントエンドで組み合わせ生成**
- 全人員計画ケース × 全シナリオを取得

### Issue #12: 浮動小数点精度エラー

**画面**: `/master/indirect-work-cases`
**カテゴリ**: バグ修正
**報告内容**: 「その他」に7と入れて保存すると、7.000000000000001と表示される

#### 現状の実装

- `IndirectWorkRatioMatrix.tsx`: UI上は0〜100%で入力
- 保存時に `ratio: ratioPercent / 100` で0〜1に変換
- DB: `DECIMAL(5,4)` で正確に保存
- 表示時: `ratio * 100` で百分率に戻す

#### 原因

JavaScript の IEEE 754 浮動小数点演算:
```javascript
7 / 100 = 0.07  // DB保存は正確（DECIMAL）
0.07 * 100 = 7.000000000000001  // JS演算の丸め誤差
```

#### 修正案

`IndirectWorkRatioMatrix.tsx` の表示値計算に丸め処理を追加:

```typescript
// Before
const displayValue = ratio * 100;

// After
const displayValue = Math.round(ratio * 10000) / 100;
// または
const displayValue = parseFloat((ratio * 100).toPrecision(10));
```

#### 影響範囲

| ファイル | 変更内容 |
|----------|---------|
| `IndirectWorkRatioMatrix.tsx` | 表示値の丸め処理追加 |

同様の問題が発生し得る画面:

| 画面 | ファイル | 百分率変換 |
|------|----------|:----:|
| `/master/indirect-work-cases` | `IndirectWorkRatioMatrix.tsx` | ○ |
| `/indirect/simulation` | `CalculationResultTable.tsx` | △（比率表示がある場合）|
